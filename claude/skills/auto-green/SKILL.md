# テスト自動修正ループ

## 概要

テスト失敗を起点に「実行 → 分析 → 修正 → 再実行」を全自動で繰り返し、全テスト Green に収束させる。

**鉄則: テストコードは正。修正するのは実装コードのみ。**

テスト自体がおかしいと判断した場合は修正せず、ユーザーに報告して判断を仰ぐ。

## ワークフロー

```
1. テスト実行方法の検出
2. テスト全件実行
3. 失敗の分類・ソート
4. 修正 → 再実行（最大5イテレーション）
5. 結果レポート
```

### Step 1. テスト実行方法の検出

実装に入る前に、プロジェクトのテスト実行方法を特定する。以下を順に探す:

1. `Makefile` — `make test` 等のターゲットがあるか
2. `pyproject.toml` — `[tool.pytest.ini_options]` や `[tool.hatch]` 等の設定
3. `package.json` — `scripts.test` の定義
4. `setup.cfg` / `pytest.ini` / `vitest.config.ts` / `jest.config.*`

見つかったコマンドを以降のループで一貫して使う。見つからなければユーザーに確認する。推測で実行しない。

**テスト対象の絞り込み:** ユーザーが特定のテストファイルやディレクトリを指定している場合はそれに従う。指定がなければ全件実行。

### Step 2. テスト全件実行

テストを実行し、出力を全文キャプチャする。

- 成功テスト数・失敗テスト数・エラー数を把握する
- 全件パスしていればここで終了。修正不要であることを報告する
- 失敗がある場合は Step 3 へ

### Step 3. 失敗の分類・ソート

失敗を以下のレベルに分類し、**L1 から順に修正する**（根本原因から潰す）:

| レベル | 種別 | 代表的なエラー | 修正戦略 |
|--------|------|----------------|----------|
| **L1** | Import / Syntax | `ModuleNotFoundError`, `ImportError`, `SyntaxError`, `IndentationError` | import パス修正、構文修正 |
| **L2** | Type / Attribute | `TypeError`, `AttributeError`, メソッド未定義 | シグネチャ修正、メソッド追加、型修正 |
| **L3** | Logic | `AssertionError`, 値の不一致、条件分岐の誤り | ロジック修正。最も慎重に扱う |
| **L4** | Environment | `FileNotFoundError`, `ConnectionError`, `PermissionError`, `TimeoutError` | **自動修正しない。即座にユーザーへエスカレーション** |

**同一レベル内の優先順位:**
- 複数テストで共通して参照されるモジュール/関数の失敗を先に直す
- 1つの修正で複数テストが通る可能性が高いものを優先する

### Step 4. 修正 → 再実行ループ

以下を **最大5イテレーション** 繰り返す:

```
4a. 最優先の失敗グループを選択
4b. 関連する実装コードを読む
4c. 根本原因を特定する
4d. 最小限の修正を実装コードに適用
4e. テスト全件再実行
4f. 結果を評価 → 継続 or 終了 or エスカレーション
```

#### 4a. 失敗グループの選択

- L1 → L2 → L3 の順に処理する
- L4 が検出されたら即座にエスカレーション
- 同一レベルで複数の失敗がある場合、共通の根本原因を持つものをグルーピングする

#### 4b〜4c. コード読解と原因特定

- テストコードから「期待される振る舞い」を読み取る
- 実装コードから「実際の振る舞い」を読み取る
- 差分が修正ポイント

**やること:**
- エラーメッセージとスタックトレースの全文を読む
- 関連するソースファイルを特定して読む
- 変更履歴が参考になる場合は `git diff` や `git log` を確認する

**やらないこと:**
- エラーメッセージだけで推測して修正しない。必ずソースを読む
- テストの意図を勝手に解釈しない。テストコードに書いてあることが仕様

#### 4d. 最小限の修正

- テストが通る最小限の変更を行う
- 修正は実装コード（`src/` 等）のみ。テストコード（`tests/` 等）には触らない
- 複数の修正方法がある場合、最もシンプルで副作用の少ないものを選ぶ

#### 4e. テスト全件再実行

修正した対象テストだけでなく、**全テストを再実行する**。

- 対象テストだけ通しても、他のテストにリグレッションが起きていれば意味がない

#### 4f. 結果評価

| 結果 | アクション |
|------|----------|
| 全テストパス | **成功。Step 5 へ** |
| 失敗が減った | 次のイテレーションへ（残りの失敗を処理） |
| 失敗が変わらない | 別のアプローチを試す。同じ修正を2回繰り返さない |
| 新しい失敗が増えた | **修正を revert し、ユーザーへエスカレーション** |
| イテレーション上限到達 | **Step 5 へ（残件報告）** |

### Step 5. 結果レポート

ループ終了後、以下を報告する:

```
## テスト修正結果

- 初期状態: X件失敗 / Y件成功
- 最終状態: X件失敗 / Y件成功
- イテレーション: N / 5

### 修正した内容
- [ファイルパス]: 修正内容の要約

### 未解決の失敗（あれば）
- [テスト名]: 失敗理由と修正できなかった原因
- 推奨アクション: （手動で確認すべきポイント）
```

## エスカレーション条件（自動修正を中断して development-partner にループバックする）

以下のいずれかに該当したら、自動修正を止めて結果レポートを出力し、development-partner にループバックする:

1. **L4（環境エラー）を検出** — ファイル不在、接続エラー等はコード修正では直らない
2. **同じテストが同じエラーで3回連続失敗** — アプローチが根本的に間違っている
3. **修正によって新しい失敗が発生** — revert してから報告
4. **5イテレーションで収束しない** — 残件を報告
5. **テストの意図が読み取れない** — テスト自体に問題がある可能性。判断をユーザーに委ねる
6. **修正が大規模になる**（新規ファイル作成、3ファイル以上の同時変更） — 影響範囲が大きいため確認

## ハードルール

- **テストコードを修正しない。** テストは仕様。実装を仕様に合わせる
- **各イテレーションでテスト全件実行する。** 対象テストだけの部分実行は禁止
- **修正前に必ずソースコードを読む。** エラーメッセージだけで修正しない
- **同じ修正を2回繰り返さない。** 効かなかった修正を再適用しても効かない
- **revert できる状態を保つ。** 修正が裏目に出たら戻せること
- **上限5イテレーション。** 超えたら development-partner にループバック

## development-partner へのループバック

エスカレーション条件に該当した場合、結果レポートを出力した上で **development-partner を自動的に発動する**（ユーザーの確認は求めない）。

**渡す情報:**

1. 失敗しているテストの一覧とエラーメッセージ
2. 試みた修正とその結果（何をやって何が起きたか）
3. エスカレーション理由（6条件のどれに該当したか）

auto-green 側で「この問題は dp で解けるか」をフィルタリングしない。判断は development-partner に委ねる。

## spec-implementer との連携

- spec-implementer で Red → Green の Green にならないとき、このスキルに切り替えて修正ループを回す
- このスキルで全テスト Green にした後、spec-implementer の Refactor フェーズに戻る
- 役割分担: spec-implementer =「新しく作る」、auto-green =「壊れたものを直す」

## 自己評価（応答の最後に必ず実行）

結果レポート（Step 5）を書き終えたら、送信前に以下を検証する。問題があれば修正してから送信する。自己評価の結果はレポート末尾に簡潔に提示する。

### チェック1: テストコード不可侵

- 修正対象が全て実装コードであるか。テストコード（`tests/` 等）に変更を加えていないか
- 違反している場合: 該当の変更を revert し、実装コード側で解決し直す

### チェック2: 修正の重複

- 同じファイルの同じ箇所に、前のイテレーションと同一の修正を再適用していないか
- 重複がある場合: そのアプローチは効かない。別の修正戦略に切り替えるか、エスカレーションする

### チェック3: エスカレーション判断

- エスカレーション6条件のいずれかに該当しているのに続行していないか
- 特に「新しい失敗が増えた」「同じエラーで3回連続失敗」を見逃していないか
- 該当している場合: 即座にループを止めてエスカレーションする

### チェック4: レベル順序の遵守

- L1 → L2 → L3 の順に修正したか。L3（ロジック）を先に触って L1（import/syntax）が残っていないか
- 順序が乱れている場合: 残っている下位レベルの失敗を先に処理する

### チェック5: レポートの完全性

- 結果レポートに以下が揃っているか: 初期状態、最終状態、イテレーション数、修正内容、未解決の失敗（あれば）
- エスカレーション時: development-partner に渡す3点（失敗一覧、試みた修正と結果、エスカレーション理由）が揃っているか

### 提示ルール

- **問題を検出した項目**: 修正内容を明記する
- **問題なしの項目**: 一行で済ますか省略
- **分量は3〜5行**を目安。結果レポートの末尾に独立セクションとして置く
