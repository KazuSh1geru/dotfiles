# 6つの思考道具 リファレンス

Link and Motivation「7つの思考道具」に準拠し、開発文脈に適用したもの。

## 選択ガイド

問題の性質に応じて思考道具を選択する。以下のシグナルを手がかりにする。

| シグナル | 推奨する思考道具 |
|---|---|
| 「情報が多すぎて整理できない」「ぐちゃぐちゃ」 | 分類・ラベリング |
| 「組み合わせを考えたい」「可能性を広げたい/絞りたい」 | マトリクス |
| 「選択肢が複数あって迷う」「どれにする？」 | 選択基準 |
| 「全体像がつかめない」「仕組みがわからない」 | 構造化 |
| 「何が原因かわからない」「なぜこうなった？」 | 切り分け |
| 「どこに注力すべきかわからない」「要素が絡み合っている」 | システム思考 |

複数のシグナルが該当する場合は、最も支配的なシグナルを優先して最大3つを選ぶ。

### 組み合わせパターン

思考道具は単体でも使えるが、組み合わせると威力が増す。よくある連携:

- **分類・ラベリング → マトリクス**: まず分類して軸を導き出し、マトリクスで組み合わせを検討
- **分類・ラベリング → 構造化**: 分類した要素を階層的に整理して全体像を把握
- **切り分け → システム思考**: 原因を特定した後、要素間の影響関係を分析してキーファクターを絞る
- **構造化 → 選択基準**: 全体像を把握してから、対策の候補を評価・選定

## 1. 分類・ラベリング

**一言**: 情報を軸で分け、名前をつけて複雑性を縮減する。

**いつ使う**: 情報や素材が多すぎて処理能力を超えているとき。問題がぐちゃぐちゃで何から手をつけていいかわからないとき。

**適用手順**:
1. 分類の目的を明確にする（何のために分けるのか）
2. 似たもの同士をグルーピングしながら軸を設定する（性質の違う複数の軸を混ぜない）
3. 分類後はグループごとに特徴を表すラベル（名前）をつける
4. 各グループに「抜け・漏れ」がないか検証する

**ポイント**: どんな軸を採用するかで分類は大きく変わる。軸の選定が全て。ラベリングがグループに新しい意味を付加する。

**出力形式**:
```
分類の目的: {何のために分けるか}
分類軸: {採用した軸とその理由}

[ラベルA]: {グループの特徴}
  - {要素1}
  - {要素2}

[ラベルB]: {グループの特徴}
  - {要素3}
  - {要素4}

[ラベルC]: {グループの特徴}
  - {要素5}

網羅性チェック: {抜け漏れがないか}
```

**開発での活用例**:
- バグレポートが大量にあるとき → 影響範囲・原因種別・緊急度で分類
- 技術的負債の棚卸し → 負債の性質（設計負債・依存負債・テスト負債等）で分類してラベリング
- 要件がぐちゃぐちゃなとき → 機能要件 / 非機能要件 / 制約条件で分類

## 2. マトリクス

**一言**: 2軸の組み合わせで「可能性の創出」または「可能性の縮減」を行う。

**いつ使う**: 「何か」と「何か」を組み合わせて新しいパターンを見つけたいとき、または想定パターンを絞り込みたいとき。

**適用手順**:
1. 何と何の組み合わせを考えるのか、目的を明確にする
2. 縦軸・横軸を設定する（必要に応じて前工程で「分類」を行い軸を導き出す）
3. 各マス目（組み合わせパターン）の中身を検討する
4. 「新しいパターンの検討」または「想定パターンの絞込み」を行う

**ポイント**: 前工程の「分類」でラベリングした結果がマトリクスの軸になる。組み合わせ意味がないものを排除して「可能性の縮減」を行うのが実務では多い。生き残った組み合わせにラベリングする。

**出力形式**:
```
目的: {何のためにマトリクスを作るか}
縦軸: {軸の定義}
横軸: {軸の定義}

|  | {横1} | {横2} | {横3} |
|---|---|---|---|
| {縦1} | {パターンor除外} | {パターンor除外} | {パターンor除外} |
| {縦2} | {パターンor除外} | {パターンor除外} | {パターンor除外} |
| {縦3} | {パターンor除外} | {パターンor除外} | {パターンor除外} |

生き残ったパターン:
- パターンA「{ラベル}」: {説明}
- パターンB「{ラベル}」: {説明}
```

**開発での活用例**:
- API設計で「リソース種別 × 操作種別」のマトリクスを作り、必要なエンドポイントを網羅
- テスト設計で「入力パターン × 状態パターン」の組み合わせからテストケースを絞り込み
- 障害対応で「影響範囲 × 発生頻度」のマトリクスで対応優先度を決定

## 3. 選択基準

**一言**: 評価項目をMUST/WANTに分け、候補案を構造的に評価・選定する。

**いつ使う**: 複数の選択肢から最適なものを選ぶ「どれにしようかな？」という場面。

**適用手順**:
1. 何のための選択を行うのかを明確にする
2. 仮の候補案（選択案）を列挙する
3. 候補案を評価する「評価項目」を列挙する（期待成果＝目的達成時の具体的な姿・数値指標 / 制約条件＝期限・コスト・人員）
4. 評価項目の優先順位付けを行う（MUST: 絶対にはずせない / WANT: 可能なら満たしたい。WANTは高・中・低で重み付け）
5. 評価項目に照らして候補案の絞込みを行う（MUST項目を満たさないものは検討対象外）
6. 評価項目に照らして新代替案を創り出す

**ポイント**: MUST を満たさない案は理由を問わず除外する。これで検討対象が一気に減る。さらに、既存案の評価で終わらず「評価項目を全て満たす新案」を考えるステップ(6)がミソ。

**出力形式**:
```
目的: {何のための選択か}

候補案:
- 案I: {概要}
- 案II: {概要}
- 案III: {概要}

評価:
| 評価項目 | 種別 | 案I | 案II | 案III |
|---|---|---|---|---|
| {項目1} | MUST | o | o | x → 除外 |
| {項目2} | MUST | o | o | - |
| {項目3} | WANT(高) | o | △ | - |
| {項目4} | WANT(中) | x | o | - |
| {項目5} | WANT(低) | o | o | - |

仮決定: {案X}（理由: ...）

新案の検討:
→ {MUST全クリア + WANTを最大化する新案があるか}
```

**開発での活用例**:
- 技術選定（ライブラリA vs B vs C）でMUST=メンテ状況・ライセンス、WANT=パフォーマンス・学習コスト
- インフラ構成の選定（ECS vs Lambda vs EKS）でMUST=コスト上限・SLA、WANT=運用負荷・スケーラビリティ
- 設計案の比較（モノリス vs マイクロサービス vs モジュラーモノリス）

## 4. 構造化

**一言**: 混沌とした情報を階層化し、水準を揃えて全体像を把握する。

**いつ使う**: 重要な要素を見落とすことなく全体像を把握したい「どういう仕組みなの？」という場面。

**適用手順**:
1. 構造化の目的を明確にする（何のために構造化するのか）
2. 分類とラベリングを行う
3. 目的に照らして分類の第一水準を決める
4. 第二水準以降の要素をさらに分類していく
5. 各要素間の水準をさらに揃える

**ポイント**: 混在しがちなもの（大きい概念 vs 小さい概念、原因 vs 結果、目的 vs 手段、表層 vs 背景）を見抜いて水準を揃える。第一水準の選び方が構造全体を決める。

**出力形式**:
```
目的: {何のために構造化するか}
第一水準の軸: {採用した分類軸とその理由}

{第一水準ラベルA}
├── {第二水準A-1}
│   ├── {第三水準A-1-a}
│   └── {第三水準A-1-b}
└── {第二水準A-2}

{第一水準ラベルB}
├── {第二水準B-1}
└── {第二水準B-2}
    ├── {第三水準B-2-a}
    └── {第三水準B-2-b}

水準チェック:
- 同一水準の要素は同じ抽象度か: {yes/no + 指摘}
- 原因と結果が混在していないか: {yes/no + 指摘}
- 目的と手段が混在していないか: {yes/no + 指摘}
```

**開発での活用例**:
- システムの全体アーキテクチャを階層で整理（インフラ → サービス → モジュール → クラス）
- 要件を構造化する（ユーザー種別 × 機能領域で第一水準を決め、各機能の詳細を下位に展開）
- 障害の影響範囲を構造化して把握

## 5. 切り分け

**一言**: 発生事象と非発生事象を比較し、原因を絞り込む。

**いつ使う**: 望ましくない問題が生じたときに原因を特定したい「どうしてこうなったの？」という場面。

**適用手順**:
1. 何の原因究明を行うのかを明確にする（文章化して思考のブレを防ぐ）
2. 問題の事実情報を整理する（事実と憶測を分ける）
3. IS（発生事象）と IS NOT（非発生事象）で切り分ける
4. IS の特性（相違点）や変化情報を整理する
5. 推定原因を複数想定する
6. 推定原因を切り分け結果で検証し、絞り込む

**ポイント**: 「起きていること」だけでなく「起きていないこと」を明確にするのがミソ。IS NOT との境界線に原因のヒントがある。推定原因は切り分けの全項目と照合して検証する（○=説明できる、×=説明できない、△=部分的）。

**出力形式**:
```
原因究明対象: {何の原因を特定するか}

切り分け:
| 観点 | IS（発生） | IS NOT（非発生） |
|---|---|---|
| WHAT: 何が | {事実} | {事実} |
| WHAT: どうした | {事実} | {事実} |
| WHERE: どこの | {事実} | {事実} |
| WHERE: どの部分 | {事実} | {事実} |
| WHEN: いつの | {事実} | {事実} |
| WHEN: どんな時 | {事実} | {事実} |
| EXTENT: どれくらい | {事実} | {事実} |
| EXTENT: 傾向/パターン | {事実} | {事実} |

ISの特性（相違点）: {IS NOT と比べて IS だけに見られる特徴}
ISに関する変化: {最近変わったこと、変化の時期}

推定原因の検証:
| 観点 | IS | IS NOT | 原因A | 原因B | 原因C |
|---|---|---|---|---|---|
| WHAT: 何が | ... | ... | o/x | o/x | o/x |
| ... | ... | ... | ... | ... | ... |

結論: {最も整合する推定原因}
```

**開発での活用例**:
- 本番障害: 「このサーバーでは起きるが、あのサーバーでは起きない」→ 差分に原因がある
- テスト失敗: 「このテストは落ちるが、似た別のテストは通る」→ 何が違うか
- パフォーマンス劣化: 「特定の時間帯だけ遅い」→ IS NOT の時間帯との差分を調べる

## 6. システム思考

**一言**: 要素を分解するだけでなく、要素間の影響関係を分析して全体最適のキーファクターを見つける。

**いつ使う**: 複数の要素が絡む問題で「で？どれに注力すれば？」という場面。

**適用手順**:
1. 何の対策を考えるのか、何の問題解決をするのかを明確にする
2. 目的達成に影響する事象を諸要素に分解する（分類を使う）
3. 分類した要素間の関係性（影響関係: +促進 / -抑制）を考える
4. 最も全体にインパクトを与える関係性を特定する（観点: 影響の大きさ、影響の広さ、影響の早さ、ダウンサイドリスク）

**ポイント**: 要素分解だけ（デカルト的な要素還元法）では「全てを同時実現することは不可能」で止まる。要素間の影響関係を見ることで「全体最適を見つけることが可能」になる。これが要素分解との決定的な違い。

**出力形式**:
```
目的: {何を達成したいか}

要素分解:
- {要素A}
- {要素B}
- {要素C}
- {要素D}

要素間の影響関係:
{要素A} --[+]--> {要素B}  （AがBを促進する）
{要素B} --[-]--> {要素C}  （BがCを抑制する）
{要素C} --[+]--> {要素D}  （CがDを促進する）
...

キーファクター: {要素X}
理由:
- 影響の大きさ: {評価}
- 影響の広さ: {評価}
- 影響の早さ: {評価}
- ダウンサイドリスク: {評価}

推奨アクション: {キーファクターに対する具体的な施策}
```

**開発での活用例**:
- 開発速度が落ちている → 要素（技術的負債、テストカバレッジ、レビュー待ち、仕様変更頻度等）を分解し、影響関係を分析してボトルネックを特定
- サービスの信頼性向上 → 要素（監視、デプロイ頻度、テスト、障害対応フロー等）の相互影響を分析
- チームの生産性改善 → 要素間のフィードバックループを可視化してレバレッジポイントを見つける
