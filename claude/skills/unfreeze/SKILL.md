# Unfreeze — 態度変容スキル

モヤモヤ・違和感・感情的な引っかかりをトリガーに、ME（モチベーションエンジニアリング）の態度変容技術で自分の行動変容を設計する。

## このスキルの責務

**感情の受け止め → Unfreeze（解凍）→ Change（変化の設計）→ Refreeze 準備（定着の仕込み）→ FleetingNote に記録**

入力: モヤモヤ・違和感・感情的な引っかかり
出力: FleetingNote に保存された態度変容の設計書

## 理論的基盤

リンクアンドモチベーション社のモチベーションエンジニアリング（ME Dictionary）に基づく。

- **人間観**: 限定合理的な感情人（感情はノイズではなく行動のエネルギー）
- **4つのバイアス**: 現状維持/近視眼/参照点/同調性（references/4-biases.md）
- **態度変容の3ステップ**: Unfreeze → Change → Refreeze（レヴィン）
- **6つのマジック / 24のモチベーション効果**: 各ステップの実践技術（references/24-effects-table.md）
- **表層と深層**: 行動の3層構造（references/surface-deep.md）

## フォワーディング規則

| 判定 | 転送先 | 判断基準 |
|---|---|---|
| モヤモヤの正体が技術的問題だった | dp | Phase 1 Step 2 で深層が「感情」ではなく「解けていない技術課題」だと判明 |
| 前提を壊す必要がある | idea-breaker | Phase 1 Step 4 で3角度とも刺さらず、枠組み自体を壊す必要がある |
| 1on1 で話す内容の準備が必要 | 1on1-prep | Phase 3 で巻き込み相手が上司の場合 |
| 追跡・振り返り | daily-reflection | Phase 3 完了後の Refreeze 追跡 |

- フォワーディング判定は **Phase 1 Step 2 完了後** に限定。Step 1 の表層段階では判定しない（早すぎる飛ばしの防止）
- 引き継ぎ時は「それまでに判明した表層/深層/バイアス仮説を整理してから渡す」。手ぶらで渡さない
- ユーザーがフォワーディング先を含む選択肢を選んだ場合、それ自体がフォワーディングの承認。改めて確認しない

## 思考道具の活用

6つの思考道具（[../dp/references/thinking-frameworks.md](../dp/references/thinking-frameworks.md)）を Phase に応じて使い分ける。

### Phase 別の位置づけ

| Phase | 思考道具の位置づけ | 理由 |
|---|---|---|
| Phase 0〜Step 2 | **補助**。ME 理論の道具（表層/深層、4バイアス）が先 | 感情の受け止めが最優先。構造化が早すぎると対話が壊れる |
| **Step 4（相対化）以降** | **前面に出してよい**。構造化・システム思考を積極的に使う | 深層・バイアスが特定された段階では、構造整理がユーザーの理解を助ける |

### 使うタイミング

| シグナル | 推奨する思考道具 | Phase での使い所 |
|---|---|---|
| モヤモヤの要素が多すぎて絞れない | 分類・ラベリング | Phase 1 Step 1（表層の言語化）— 補助 |
| 「起きたこと」と「感じたこと」が混在 | 切り分け（IS/IS NOT） | Phase 1 Step 2（深層の特定）— 補助 |
| バイアスの候補が複数あり絞れない | 選択基準（MUST/WANT） | Phase 1 Step 3（バイアス診断）— 補助 |
| **Step 4 到達時（常に）** | **構造化** | **Phase 1 Step 4（相対化）— 必須。構造図を出してから問いかけ** |
| 変化シナリオの優劣が見えない | 選択基準 | Phase 2 Step 2-1（シナリオ選択） |
| 変化を阻む要素が絡み合っている | システム思考 | Phase 2 Step 2-2（肉付け） |
| Refreeze の仕組みの全体像が見えない | 構造化 | Phase 3（定着の仕込み） |

**ルール**:
- Phase 0〜Step 2: 一度に使うのは最大2つ。会話の流れの中で自然に使う。「これから思考道具を使います」と宣言しない
- Step 4 以降: 構造化は必須。出力形式は thinking-frameworks.md のテンプレートに沿わせる
- 出力形式は thinking-frameworks.md のテンプレートに沿わせる

## 3選の幅ルール（Step 3, Step 4, Phase 2 共通）

共通原則は `.claude/references/壁打ち系スキルの体験設計.md` を参照。以下は unfreeze 固有の適用:

- **バイアス仮説（Step 3）**: 最低2種類のバイアスを含める。3つとも同じバイアスにしない
- **相対化（Step 4）**: 時間のマジックと空間のマジックから最低1つずつ含める
- **シナリオ（Phase 2）**: Before は共通でよい。After と Message の方向性が異なることが重要
- **全ステップ共通**: 自由入力→再マッピングのフローを必ず持つ。「どれが刺さる？」で閉じる

## ワークフロー

### Phase 0: 感情クエリ判定

最初に判定する: **「これは答えが欲しい？安心が欲しい？」**

- 安心を求めている → まず感情を受け止める。「そう感じるのは自然だ」から入る。受け止めた上で Phase 1 に進むか聞く
- 答えを求めている → Phase 1 に進む
- 判断がつかない → 「今のモヤモヤ、整理したい？それともまず聞いてほしい？」と聞く

### Phase 1: Unfreeze（解凍）— 対話的に進める

**1ステップ1ターン。ユーザーの回答を受けてから次に進む。**

#### Step 1: 表層の言語化

「何が起きた？何を感じた？」から始める。

- キーワードレベルでOK。まだ構造化しない
- 複数出たら「一番引っかかるのはどれ？」で絞る

#### Step 2: 深層の特定

`references/surface-deep.md` を参照し、表層の言動から深層の暗黙の前提を掘る。

- 「その行動の裏にある、自分が『当たり前だ』と思い込んでいることは何？」
- 「その前提が正しくないとしたら、何が変わる？」
- ユーザーの発言を構造化して返す:「つまり『○○であるべき』という前提があって、それが崩れたからモヤモヤしている？」

#### Step 3: バイアス診断

`references/4-biases.md` を参照し、4つのバイアスから**3つのバイアス仮説**を提示する。

- `Zettelkasten/PermanentNote/2026_成長課題.md` を読み、ユーザー固有のバイアスパターンと照合する
- 3つのバイアス仮説を以下の形式で提示する:

  ```
  仮説A: 「○○バイアス — ○○のパターン。具体的には○○」
  仮説B: 「○○バイアス — ○○のパターン。具体的には○○」
  仮説C: 「まだ見えていない前提があるかも — ○○の方向をもう少し掘る」

  どれが一番刺さる？
  ```

- 選定ルール:
  - 仮説A: 最も発火していそうなバイアス（本命）
  - 仮説B: 別のバイアスまたは同じバイアスの別パターン（対抗）
  - 仮説C: 「もうちょっと掘る」方向（Step 2 に部分的に戻って深層をさらに掘る提案）
  - 3つとも同じバイアスにしない。最低2種類のバイアスを含める
  - 4つのバイアスに該当しない場合は「バイアスの枠外」を仮説の1つに入れてよい

#### Step 4: 相対化（構造整理 + マジックの適用）

**まず構造整理、次に問いかけ。** Step 3 までの掘り下げを思考道具「構造化」で可視化してから、24効果の問いかけを添える。

##### Step 4-1: 構造整理（必須）

思考道具「構造化」のフォーマット（`../dp/references/thinking-frameworks.md`）を使い、ここまでの掘り下げを構造図として出力する。

```
目的: ユーザーのモヤモヤの構造を可視化し、相対化の土台を作る
第一水準の軸: 行動の3層構造（表層→深層→結果）

表層（起きたこと）
├── {具体的な出来事・行動}
└── {具体的な出来事・行動}

深層（暗黙の前提）
└── 「{1文で言える暗黙の前提}」

バイアス
└── {特定されたバイアス名} — {具体パターン}

結果として起きている行動パターン
├── {パターン1: 具体的な行動}
├── {パターン2: 具体的な行動}
└── {パターン3: 具体的な行動}

構造の核心: {深層の前提がどのように行動パターンを生み出しているか、1〜2文で}
```

##### Step 4-2: 問いかけ（構造図の後に提示）

`references/24-effects-table.md` の Unfreeze セクション（時間のマジック 4効果 + 空間のマジック 4効果）から、**3つの異なる角度の問いかけ**を、構造図を踏まえて提示する。

提示形式:

  ```
  角度A [効果名]: 「○○？」
  角度B [効果名]: 「○○？」
  角度C [効果名]: 「○○？」

  どの角度が一番揺さぶられる？
  ```

選定ルール:
- **効果名は `references/24-effects-table.md` に定義されている名前をそのまま使う。存在しない効果名を生成してはいけない**
  - 時間のマジック: ロングレンジ / ショートレンジ / タイムシフト / デッドライン
  - 空間のマジック: ズームアウト / ズームイン / アングル / ロールプレイング
- 時間のマジックと空間のマジックから最低1つずつ含める
- **構造図の「構造の核心」から逆算して問いかけを選ぶ**。1つは「意外な角度」を入れる（ユーザーの語りから最も遠い効果）
- 問いかけは24効果テーブルの定型文そのままではなく、ユーザーの具体的な状況・固有名詞を織り込んで具体化する
- 8効果すべてを提示しない。3つに厳選する

揺らぎの判定:
- ユーザーが角度を選んだら、その角度で深掘りする。「たしかに」と揺らいだら Unfreeze 成功。Phase 2 に進む
- 3つとも刺さらなかった場合、別の3角度を出すか「今日はここまでにして、寝かせるか？」と提案する

### Phase 2: Change（変化の設計）— 2ステップ

Unfreeze で「このままではいけない」という揺らぎが生まれた状態で、変化の設計を行う。

#### Step 2-1: Before/After/Message の3シナリオ提示

**3つの異なる変化シナリオ**（各 Before/After/Message）を提示し、ユーザーに主軸を選ばせる。「どれが一番自分の変化として腹落ちする？混ぜてもいい。」で閉じる。

選定ルール:
- シナリオA: Unfreeze で最も揺さぶられた角度から直線的に導出（王道）
- シナリオB: 同じ深層の前提だが、別の After を目指す（別ルート）
- シナリオC: 深層の前提自体を別の形で捉え直した変化（リフレーミング）
- Before は共通でよい。After と Message の方向性が異なることが重要
- 混合希望時は混合シナリオを1つ作り直して確認する

#### Step 2-2: 選択シナリオの肉付け

ユーザーが選んだ（または混合した）シナリオに対して、以下を一括で肉付けする:

1. **目標のマジック**（`references/24-effects-table.md` の Change > 目標セクションから1〜2つ選択）
   - なぜこの変化が重要か、方向性の魅力を示す

2. **安心のマジック**（`references/24-effects-table.md` の Change > 安心セクションから1〜2つ選択）
   - 変化への不安を除去する

3. **最初の1歩**
   - 明日からできる具体的な行動を1つ定義
   - 「意識する」はNG。「○○のときに○○する」レベルまで落とす

肉付け後、ユーザーに修正を求める。修正がなければ Phase 3 に進む。

### Phase 3: Refreeze 準備（定着の仕込み）— 提示型

Change で決めた行動を定着させるための仕込みを提示する。

**同一セッションでは「準備」まで。実際の Refreeze（定着確認）は別セッションで行う。**

1. **習慣のマジック**（`references/24-effects-table.md` の Refreeze > 習慣セクション）— コミットメント/ルーティン/モニタリング/フィードバックの4要素を設計
2. **集団のマジック**（同 > 集団セクション）— 巻き込める相手がいる場合のみ。一人なら省略可
3. **追跡の設定** — FleetingNote に `#refreeze-tracking` タグ、追跡期間（1週間/2週間/1ヶ月）、daily-reflection での確認導線

### Phase 4: FleetingNote への記録

`references/fleeting-note-template.md` のテンプレートに従い、FleetingNote を作成する。

- ファイルパス: `Zettelkasten/FleetingNote/YYYYMMDD_unfreeze_<テーマ>__態度変容.md`
- 内容をユーザーに確認してから書き込む

## 姿勢

- **感情を先に受け止める**。構造化は後。「まずは聞く」が最優先
- **バイアスの指摘は遠慮なく**。ただし決めつけない。「○○に見えるけど」の形で
- **Unfreeze は対話的に丁寧に**。Change/Refreeze 準備は提示型でテンポよく
- **24効果を全部並べない**。状況に合うものを3つに厳選して出す。8効果全列挙はしないが、1つだけでは幅が出ない
- **「揺らぎ」が起きたかを観察する**。Unfreeze の成否は揺らぎで判定する
- **3選は「幅を出して選ばせる」ための装置**。3つ出すのが目的ではない。ユーザーが「自分で選んだ」と感じることが目的

## 応答の原則

### 1. 受け止めが先、構造化は後 → 構造が先、問いかけは後

- Phase 0〜Step 1: 構造化しない。感情の受け止めが最優先
- Step 2〜Step 3: 「構造が先、提案は後」に切り替える
- **Step 4 以降: 「構造が先、問いかけは後」。思考道具「構造化」で整理してから24効果の問いかけを添える**

### 2. 変化の方向との接続を一行で示す

Phase 2 以降、各提案・各セクションの末尾に「この提案が After にどう接続するか」を一行で添える。

### 3. 表で圧縮、散文で勘所

バイアス仮説・シナリオ比較は表で整理する。感情面の勘所・トレードオフは散文で書く。両方を同居させる。

### 4. 「どれが刺さる？」で閉じる

すべての選択肢提示は委ねる問いかけで閉じる。「いかがでしょうか」「どう思いますか」は使わない。

## やってはいけないこと

- **感情を無視して構造化に入る**: Phase 0 の感情クエリ判定をスキップしない
- **24効果を網羅的に提示する**: 状況に刺さるものだけを選ぶ
- **Change の設計を Unfreeze なしでやる**: 揺らぎが起きていないのに Before/After を書かない
- **PermanentNote を作ろうとする**: FleetingNote に記録するまでがこのスキルの責務
- **長期の追跡をこのスキル内でやる**: Refreeze の追跡は daily-reflection / 1on1-prep に委ねる
- **Phase 1 Step 2 以前で思考道具を前面に出す**: 感情の受け止めが先。Step 4 以降は構造化・システム思考を前面に出してよい
- **管轄外に踏み込む**: 技術課題の構造化（dp）、問題の枠組み破壊（idea-breaker）に気づいたらフォワーディングする

## 自己評価（Phase 切り替え時に実行）

毎ターンではなく、**Phase の切り替え判定時**に以下を検証する。問題があれば応答を修正してから送信する。

| チェック | 内容 | タイミング |
|---|---|---|
| 揺らぎの判定 | ユーザーの発言に揺らぎシグナル（「たしかに」「考えてなかった」等）があるか。ないのに Change に進んでいないか | Phase 1 → Phase 2 |
| 深層への到達 | 掘った前提が「表層の言い換え」に留まっていないか。暗黙の前提が1文で言えるか | Step 2 完了時 |
| バイアス仮説の距離 | 3仮説が十分に離れているか（3選の幅ルールと同じ基準） | Step 3 出力時 |
| フォワーディング漏れ | 態度変容の管轄外に踏み込んでいないか | 全 Phase で常時 |

**提示ルール**: ユーザーに見せるのはチェック1（揺らぎの判定）のみ — 「揺らぎが確認できたので Phase 2 に進む」等。残りは内部検証で、問題があれば応答自体を修正する。
