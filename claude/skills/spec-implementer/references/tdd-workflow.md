# TDD ワークフロー詳細

## Red-Green-Refactor サイクル

### Red: テストを書く

テストは「仕様の表明」。実装を知らない状態で書く。

言語固有のテスト例・命名規則は各言語リファレンスを参照。

### Green: 最小実装

テストを通す最短コードを書く。ハードコードでもいい。
次のテストを足す → 落ちる → バリデーション追加 → 通す → 繰り返し。

言語固有の実装例は各言語リファレンスを参照。

### Refactor: 整理する

テストが全部緑の状態で:
- 重複排除
- 命名改善
- パターン抽出（VO化、関数分離等）

**やっていいこと:** 内部構造の変更、リネーム、共通化
**やってはいけないこと:** 振る舞いの変更（テストが壊れる変更）

## テスト戦略

### テストピラミッド

```
        /  E2E  \        ← 少数。外部接続、API全体の動作確認
       /----------\
      / Integration \    ← 中程度。ユースケース + fake Repository
     /----------------\
    /    Unit Tests     \ ← 大量。VO, Entity, Domain Service
```

### Fake Repository パターン

Repository のインターフェースに対して、テスト用のインメモリ実装を作る。

言語固有の実装例・統合テストでの使い方は各言語リファレンスを参照。

## テスト作成時の注意

- **1テスト1アサーション** を基本にする。関連するアサーションのグループは許容
- **テスト間の依存禁止**。各テストは独立して実行できること
- **テストデータはテスト内で作る**。共有フィクスチャは最小限に
- **マジックナンバー禁止**。テストでも意図が読める値を使う
- **モックは最小限**。Fake Repository > モック。振る舞いの検証にはモック、状態の検証には Fake
