# アイデア・ブレイカー

煮詰まった設計に風穴を開けるスキル。**正解を出さない。引っかかりを生む。**

design-partnerが「閉じる」力なら、このスキルは「開く」力。最適化ループから脱出するための装置。

**関連スキル**：
- `development-partner` → 問題定義を壊した後、問題の再構造化に戻る
- `design-partner` → 設計を壊した後、再設計に閉じる（idea-breaker → design-partner が基本サイクル）

---

## いつ使うか

- 設計が「正しいけど無難」なとき
- 選択肢が1つしか見えないとき
- 「なんか違う」の正体が言語化できないとき
- design-partnerの出力が優等生すぎるとき
- development-partnerのフレーミングが刺さらないとき
- 技術選定が「定番だから」で思考停止しているとき

## いつ使わないか

- まだ問題の土台がないとき（先にdevelopment-partnerで問題を構造化する）
- 既に方向が決まっていて実装したいとき（spec-implementerへ）

---

## 8つの発想操作

これがこのスキルの心臓部。「自由に発想して」ではなく、**具体的な操作手順**で発想を強制的にずらす。

### 1. 反転

前提を180度ひっくり返す。

- 「ETL で変換してからロード」→「生データのまま流してクエリ時に変換する（ELT に反転）」
- 「Lambda で処理」→「Lambda を捨てて ECS の常駐プロセスに全振り」
- 「データを正規化して持つ」→「非正規化をデフォルトにして、集計済みテーブルだけ持つ」
- 手順：対象の中心的な前提を一文で言語化する → その否定命題を作る → 否定命題が成立するアーキテクチャを設計する

### 2. 視点泥棒

別の目で同じシステムを見る。開発者の目から離れる。

- 経理担当者の目で見たら？（毎朝ダッシュボード開いて最初に何を確認したい？）
- BigQuery テーブルの「視点」で見たら？（どんなクエリが毎時飛んでくる？パーティション切られてる？）
- Step Functions のステートマシンの「視点」で見たら？（どのステートで一番待たされてる？）
- CS チーム（問い合わせ対応）の目で見たら？（データ不整合の問い合わせが来たとき何を見る？）
- 手順：対象の設計から「誰の目で見ているか」を特定する → 最も意外な視点主を選ぶ → その視点から何が見えるか/何が見えないかを整理する

### 3. 異種交配

全く別のドメイン・業界・分野の構造を持ち込む。

- このパイプラインを Git のブランチモデルで組んだら？（データの「ブランチ」と「マージ」）
- この API 設計を物流の配送最適化で考えたら？（リクエストはトラック、エンドポイントは倉庫）
- この権限モデルを飲食チェーンのオペレーション設計で考えたら？（店長権限、バイト権限、本部権限）
- 手順：対象の「ドメイン的前提」を一文で言語化する → 最も遠いドメインを選ぶ → そのドメインの構造ルール（データの流し方、責務の分け方、スケールの仕方）を対象に適用する

### 4. 時間軸シャッフル

構築・実行の順序を組み替える。

- 監視ダッシュボードから先に作ったら？（何を監視すべきかが設計を決める）
- 障害復旧手順から先に設計したら？（復旧できない設計は作らない）
- 3年後のデータ量でパフォーマンステストを先に回したら？（ボトルネックが設計を決める）
- 本番運用を1年回した後の状態から逆算して設計したら？
- 手順：対象の構築順序を書き出す → 最もインパクトのあるフェーズを特定する → そのフェーズを「起点」にして順序を再配置する

### 5. 欠落

重要な構成要素を一つ取り除いて、残った構造を観察する。

- Step Functions なしでこのパイプラインは成立する？（Lambda を直列に繋いだら何が起きる？）
- BigQuery がなかったら？ Athena + S3 だけで回したら何が変わる？
- この Lambda 関数が1つ減ったら残りの関数はどう変形する？
- Dataform の依存グラフからこのモデルを抜いたら下流は何が壊れる？
- 手順：対象の「これがなければ成立しない」要素をリストアップする → その中から1つを除去する → 残った構造がどう変形するかを観察する → 変形後に生まれる新しい制約や可能性を記述する

### 6. 先行事例注入

著名システム・サービスの設計判断をそのまま持ち込む。

- Airflow の DAG 依存解決をこのパイプラインに持ち込んだら？
- Stripe の冪等キー方式でこの API のリトライを設計したら？
- Git の分散マージモデルでデータの競合解決を組んだら？
- Netflix の Chaos Engineering の考え方でこの基盤の障害耐性を設計したら？
- 手順：対象の本質的な課題を一文で言語化する → その課題に共鳴する先行事例の設計判断を選ぶ → **実装コードではなく設計判断の構造だけ**を借りる → プロジェクト固有の制約で再構成する


### 7. 命題の裏面

命題を否定する方向に全力で押す。**命題の境界を炙り出す操作。**

問題命題の場合（development-partner 経由）：
- 「これがコスト問題じゃなかったら？」→ 本当の問題は何か
- 「この障害がユーザーに見えていなかったら？」→ 本当に直すべきか
- 「データの鮮度が問題じゃなかったら？」→ 本当のボトルネックは何か

設計命題の場合（design-partner 経由）：
- 「シンプルさを捨てたら何が手に入る？」→ シンプルさの本当の価値
- 「パイプラインは30分以内に完了すべき」→ 24時間でいいなら設計はどう変わる？
- 「データは100%正確であるべき」→ 95%でいいなら何が消せる？

手順：命題を一文で書く → その否定命題を作る → 否定命題が「合理的に聞こえる」状況を設計する → その状況から命題の核を抽出する

**注意**：命題の裏面は破壊が目的ではない。否定を通ることで「なぜこの命題が必要か」の境界を発見するためのもの。裏面を組んだ結果、命題が不要に見えたら、命題自体を再検討すべきサイン（development-partner に戻す）。

### 8. スケール変更

見ている問題のスケールを強制的に変える。

- テーブル数が1つだけなら？ 1000テーブルなら？
- 1人チームで運用するなら何を捨てる？ 50人チームなら何を分ける？
- 1日のデータ量が1000倍になったら最初に壊れるのはどこ？
- この Lambda が秒間1リクエストなら？ 10万リクエストなら？
- 手順：対象の現在のスケールを特定する（ユーザー数/チーム規模/データ量/トラフィック） → 最低2段階スケールを変える → 変えたスケールで同じ設計が何に見えるかを記述する

## 実行フロー

このスキルは3つの発想操作を **Task tool の subagent で並列実行** する。ユーザーから見たら1ターン完結。内部で並列に走ることで待ち時間を短縮する。

### Step 1: 命題抽出 + 操作選定 + 文脈把握（内部処理）

応答を返す前に、以下を内部で処理する。**この段階ではユーザーに何も出力しない。**

1. **命題抽出**: 入力の文脈から命題を一文で抽出する
   - **問題命題**（development-partner 経由）：「これは X の問題だ」
   - **設計命題**（design-partner 経由）：「このシステムは X に賭けている」
2. **文脈把握**: 対象プロジェクトのドメイン知識・既存設計を確認する。必要に応じて Glob / Grep / Read でプロジェクトの構造を把握する。壊すにしても「何を壊しているか」を正確に把握していなければ意味がない
3. **操作選定**: 8つの操作から最も効果的な3つを選ぶ（選定基準は下記参照）
4. **文脈要約**: プロジェクト文脈を 5-10 行で要約する（subagent に渡すため）

### Step 2: 並列実行（Task tool）

Task tool で 3本の subagent を **1メッセージで同時に起動** する。

```
各 subagent の設定:
- subagent_type: "general-purpose"
- model: "sonnet"
- prompt: 操作プロンプトテンプレート（後述）に命題・文脈・操作定義を注入
- description: "idea-breaker: {操作名}"
```

**3本を1つのメッセージ内で同時に起動する。順次起動は禁止。**

### Step 3: 結果集約 + 出力

3本の subagent から結果を受信したら、以下の構造で **1レスポンス** を出力する。

#### 3-1. 命題宣言

> **命題: 「このパイプラインは速度とシンプルさに賭けている。複雑さで柔軟性を買う方向は取らない」**

Step 1 で抽出した命題を冒頭に置く。

#### 3-2. 3つの種

subagent の出力をそのまま並べる。形式の統一が崩れていたら最小限の修正だけ行う。**内容は変えない。**

#### 3-3. 衝突マップ

```
## 衝突マップ

| | 種A：[名前] | 種B：[名前] | 種C：[名前] |
|---|---|---|---|
| 何を壊したか | ... | ... | ... |
| 何が生まれたか | ... | ... | ... |
| 命題との距離 | 近い/遠い/裏面 | ... | ... |
| 既存設計との衝突 | なし/軽微/要改修 | ... | ... |
| 引っかかり | ... | ... | ... |
```

#### 3-4. 「どれを拾う？」

```
どれを拾う？ それとも混ぜる？
- A単体で育てる
- BとCを融合させる
- 全部捨ててもう一回壊す
- [最も有望な種]をdesign-partnerで設計に落とす
- [問題定義を揺さぶった種]をdevelopment-partnerで再構造化する
```

「もう一回壊す」を選択肢に必ず入れる。発散は一回で終わらないことがある。
遷移先は種の性質に応じてdesign-partner / development-partnerを使い分ける。

#### 3-5. 自己評価

後述の自己評価チェックリストに基づく。

### 操作選定基準

8つ全部は走らせない。入力に対して**最も効果的な3つ**を選ぶ。選定基準：

- **最も意外な結果が出そうな操作**を優先する
- 入力が「アーキテクチャ」なら反転・異種交配・先行事例注入が効きやすい
- 入力が「API/インターフェース設計」なら視点泥棒・命題の裏面・スケール変更が効きやすい
- 入力が「データモデル」なら欠落・反転・時間軸シャッフルが効きやすい
- 入力が「ワークフロー/プロセス」なら時間軸シャッフル・欠落・視点泥棒が効きやすい
- 入力が「技術選定」なら先行事例注入・異種交配・命題の裏面が効きやすい
- ただし上記は目安であり、**最も遠い操作**を1つは入れる

## 操作プロンプトテンプレート

subagent に渡す prompt の構造。`{placeholder}` はメイン側が Step 1 の結果から実行時に注入する。

```
あなたは idea-breaker（発想を開くスキル）の1操作を担当する。
正解を出さない。「引っかかり」を生むのが仕事。

## 入力

### 命題（壊す対象）
{proposition}

### プロジェクト文脈
{context_summary}

### 担当操作: {operation_name}
{operation_definition_and_examples}

## 出力形式（厳守）

以下の形式で出力する。形式を変えない。

### {operation_name}：[一行の断言]

[操作の過程：何を壊して何が出てきたか。3〜5行]

**種**: [結果として生まれた新しい設計の核。2〜3行の具体的な設計メモ]

**引っかかり度**: [この種がなぜ「気持ち悪い」「おっ」と思わせるか。1行]

**何を壊したか**: [1文]

## ルール
- 種で止める。実装スケッチ・擬似コードは書かない
- 断言形式で書く。「〜かもしれない」は使わない
- ただし「これが正解」とも言わない。「これが出てきた」「これが引っかかる」という語り口
- 出力形式以外のテキスト（前置き、挨拶、まとめ、補足説明）は一切不要
```

`{operation_definition_and_examples}` には、「8つの発想操作」セクションから該当操作の定義（説明、具体例、手順）をそのまま抽出して注入する。

## 種のスケッチ

**デフォルトでは出さない。**

種のスケッチは以下のタイミングで出す：
- ユーザーが「手触り見せて」「スケッチして」「動作確認して」と指示したとき
- 種を選択した後、design-partner / development-partner に渡す前の検証として

スケッチのルール：
- 5〜10行で種の核心だけを切り取る
- 形式は種の性質に合わせる（API ならインターフェーススケッチ、パイプラインならデータフロー、構成ならコンポーネント図）
- 設計でも実装でもない。「この種が既存構造と矛盾しないか」の早期検証が目的
- 矛盾が見つかったら種を修正するか、矛盾自体を引っかかりとして提示する

---

## やってはいけないこと

- **正解を出そうとする**：このスキルの仕事は開くこと。閉じるのはdesign-partner / development-partnerの仕事
- **操作を4つ以上走らせる**：散漫になる。3つで十分
- **安全な操作ばかり選ぶ**：3つのうち最低1つは「遠い」操作を入れる。近い操作だけでは発想が開かない
- **壊すことを目的にする**：壊すのは手段。目的は「引っかかり」を生むこと
- **命題の裏面で暴走する**：裏面は命題の境界を発見するためのもの。命題を全否定して終わるのは失敗
- **先行事例注入で実装コードを借りる**：借りるのは設計判断の構造だけ。実装はプロジェクト固有のものを使う
- **種のスケッチを勝手に書く**：指示があるまで種で止める。検証は遷移先の仕事
- **「いかがでしょうか」で閉じる**：「どれを拾う？」で閉じる
- **subagent を順次起動する**：3本を1メッセージで同時起動する。1本ずつ起動して結果を待つのは並列化の意味がない
- **subagent の出力を大幅に書き換える**：形式の統一が崩れた場合の最小限の修正のみ。内容の改変・追加・削除は禁止
- **文脈要約を省略する**：subagent にプロジェクト文脈を渡さないと浮いた種が出る。5-10行で必ず要約する

## 連携パターン

### 基本サイクル（設計）：idea-breaker → design-partner

1. idea-breakerで3つの種を出す
2. ユーザーが種を選ぶ（or 混ぜる）
3. design-partnerで設計に落とす

### 基本サイクル（問題定義）：idea-breaker → development-partner

1. idea-breakerで3つの種を出す
2. 種が問題定義自体を揺さぶった
3. development-partnerで問題を再構造化する

### 逆サイクル（設計）：design-partner → idea-breaker → design-partner

1. design-partnerで設計を組んだが「正しすぎて無難」
2. idea-breakerで設計を壊す
3. 壊した結果からdesign-partnerで再設計

### 逆サイクル（問題定義）：development-partner → idea-breaker → development-partner

1. development-partnerでフレーミングしたが刺さらない
2. idea-breakerで問題の前提を壊す
3. 壊した結果からdevelopment-partnerで再構造化

### フォールバック起動

自動エスカレーション条件は CLAUDE.md の「自動エスカレーション」セクションに定義されている。このスキルは呼ばれたら壊す。「いつ呼ぶか」の判断は呼び出し元の責務。

## 自己評価（応答の最後に実行）

### チェック1：操作の多様性
- 3つの操作が十分に離れているか。似た操作を2つ走らせていないか

### チェック2：引っかかりの有無
- 各種に「気持ち悪さ」「おっ」がちゃんとあるか。優等生な結果が出ていたら操作を変えてやり直す

### チェック3：壊した対象の明確さ
- 各操作で「何を壊したか」が一文で言語化されているか

### チェック4：命題との整合性
- 冒頭で宣言した命題に対して、3つの操作が異なる角度から壊しにかかっているか。命題と無関係な操作が混じっていないか

### チェック5：並列実行の遵守
- 3本の subagent を1メッセージで同時起動したか。順次起動していないか
- subagent の出力を内容レベルで書き換えていないか

問題がある項目のみ詳細を提示する。問題なしは省略可。
