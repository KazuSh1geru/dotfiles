# Refine Idea

フィードバック（競合調査・批評・水平思考の結果）を受けて、アイデアの改善版を作成する。

## 入力

- `$ARGUMENTS`: 改善対象のファイルパスと、フィードバックのファイルパス
- 例: `/refine-idea ideas/problems/jtc-evaluation.md` （同ディレクトリの関連フィードバックを自動で探す）
- 例: `/refine-idea ideas/solutions/peer-review.md feedback.md`

## Workflow

### Phase 1: 統合（Synthesis）
1. 元のアイデアを読み込む
2. 関連するフィードバックを収集
   - 競合調査結果（`*-competitors.md`）
   - クリティカル分析結果（`*-critique.md`）
   - 水平思考結果（`*-lateral.md`）
3. フィードバックの要点を整理

### Phase 2: 評価（Evaluation）
1. 各フィードバックの重要度を評価
2. 取り入れるべきものを選別
3. トレードオフを明確化

### Phase 3: 改善（Improvement）
1. 元のアイデアを改善
2. 新しい要素を追加
3. 弱点への対処を組み込む
4. **元の残課題はフィードバック改善の対象にしない**（下記ルール参照）

### Phase 4: 文書化（Documentation）
1. 改善版を適切なテンプレートで作成
2. 変更点を明記
3. 残課題を整理（元の残課題 + 改善で新たに生じた残課題）

### 残課題の扱いルール

元のアイデアの「残課題・未検討事項」は、まだ十分に議論・検証されていない論点である。
フィードバック（競合調査・批評・水平思考）から得た知見で改善するのは**アイデア本体**であり、残課題はフィードバック改善の対象外とする。

**やること:**
- 元の残課題をそのまま改善版に引き継ぐ
- フィードバックを踏まえて、各残課題を今後詰めるための**論点**として整理する（例: 「何を検証すればよいか」「誰に聞けばわかるか」「どんな条件で判断するか」）

**やらないこと:**
- フィードバック結果をもとに残課題を「解決済み」にする
- 残課題の内容をフィードバックで書き換える

## フィードバック統合の原則

### 競合調査からの学び
- 競合の強みから取り入れられるもの
- 競合の弱みを自分の強みにできるか
- 差別化ポイントの再定義

### クリティカル分析への対応
- 致命的な弱点への対処（必須）
- 重要な弱点への対処（推奨）
- 未検証の前提の検証計画

### 水平思考からの取り込み
- 元のアイデアを強化する要素
- 新しい切り口・視点
- 将来の拡張可能性

## 出力フォーマット

```markdown
## アイデア改善レポート

### 改善対象
[元のアイデアへのリンクと概要]

### 参照したフィードバック
- [競合調査: リンク]
- [クリティカル分析: リンク]
- [水平思考: リンク]

---

### 主要な改善点

#### 1. [改善点1のタイトル]
**元の状態**: [before]
**改善後**: [after]
**理由**: [なぜこの改善が必要か]

#### 2. [改善点2のタイトル]
[同様の形式]

---

### 取り入れなかったフィードバック
| フィードバック | 理由 |
|--------------|------|
| [内容] | [取り入れなかった理由] |

---

### 残課題

#### 元のアイデアから引き継いだ残課題
（フィードバック改善の対象外。論点のみ整理）

- [ ] [元の残課題1] → 論点: [何を検証すればよいか / 誰に聞けばわかるか / どんな条件で判断するか]
- [ ] [元の残課題2] → 論点: [同上]

#### 今回の改善で新たに生じた残課題
- [ ] [改善によって生まれた新しい課題・トレードオフ]

---

### 改善版ドキュメント

[以下に改善版の完全なドキュメントを出力]
```

## 改善版ドキュメントの作成

元のドキュメントと同じテンプレートを使用:
- 課題仮説 → `templates/problem-hypothesis.md`
- ソリューション仮説 → `templates/solution-hypothesis.md`
- プロダクトアイデア → `templates/product-idea.md`

改善版は元ファイルを**上書き**する（gitで履歴管理）

## 注意事項

- 全てのフィードバックを取り入れる必要はない
- トレードオフを明確にした上で判断
- 「改善」が「複雑化」にならないよう注意
- 核となる価値提案を見失わない
- 改善の理由を必ず記録
