# spec-reviewer

他人が書いた spec を構造化された基準で読み、具体的なフィードバックを返すスキル。
spec-writer が「書く」スキルなら、spec-reviewer は「読む」スキル。
属人的な「なんとなく良さそう」レビューを排除し、再現可能な品質チェックを行う。

---

## トリガー

- 「specをレビューして」「specのレビューお願い」等、spec レビューの明確な指示でのみ発動
- 「品質チェック」「フィードバックください」等の曖昧な指示では発動しない

---

## ハードルール（例外なし）

1. **spec の書き直しはしない。** 指摘と改善案を出すだけ。修正はspec作成者の責務。
2. **「曖昧です」で終わらせない。** 必ず具体的に指摘する。
   - NG: 「エラーハンドリングが曖昧です」
   - OK: 「FR-3のAC3.1で変換失敗時の動作が未定義。リトライ/スキップ/中断のどれか明記が必要」
3. **全ファイルを読んでからレビューを開始する。** 部分読みで指摘を始めない。
4. **存在するフェーズのみレビューする。** 未作成フェーズについては「未作成」と記録するだけで、作成を促さない。

---

## レビュー手順

```
1. spec フォルダの特定と全ファイル読み込み
2. フェーズごとに評価基準で判定
3. フェーズ横断トレーサビリティチェック
4. 指摘を 致命的 / 改善推奨 / 軽微 に分類
5. 構造化レビューレポートを出力
```

### Step 1: 対象 spec の特定と読み込み

1. **対象 spec の特定**（以下の優先順で決定する）:
   a. ユーザーが spec 名を指定した場合 → そのまま使う
   b. 指定がない場合 → 自動特定する:
      - `.spec-workflow/specs/` 配下の spec 一覧を取得する
      - 各 spec に対して `mcp__spec-workflow__spec-status` を呼び、進行中（未完了フェーズ or 未完了タスクがある）の spec を探す
      - 進行中の spec が **1つ** → 自動選択し、ユーザーに「{spec名} をレビューする」と伝える
      - 進行中の spec が **複数** → AskUserQuestion で選択させる
      - 進行中の spec が **ない** → AskUserQuestion で対象を確認する
2. `.spec-workflow/specs/{name}/` 配下の全ファイルを読み込む
   - `requirements.md`
   - `design.md`
   - `test-design.md`
   - `tasks.md`
3. steering docs があれば読み込む（`.spec-workflow/steering/*.md`）
4. 存在しないファイルは「未作成」として記録する

### Step 2: フェーズ別評価

各フェーズの評価基準（後述）に照らして判定する。

### Step 3: フェーズ横断トレーサビリティチェック

AC → Design → Test → Task の追跡を行い、漏れを検出する（後述）。

### Step 4: 重要度分類

| 重要度 | 基準 | 例 |
|--------|------|-----|
| **致命的** | 実装フェーズで手戻りが確実に発生する | ACのテストカバレッジ漏れ、未定義のエラーパス、要件とデザインの不整合 |
| **改善推奨** | 品質を上げるが、実装は可能 | 型定義の不足、図の欠如、NFRの定量化不足 |
| **軽微** | あると良いが無くても進める | 用語の揺れ、フォーマットの不統一、typo |

### Step 5: レビューレポート出力

後述のフォーマットで出力する。

## フェーズ別評価基準

### Requirements

| # | チェック項目 | 致命的になる条件 |
|---|-------------|----------------|
| R-1 | 全ACが WHEN/THEN 形式で検証可能か | 検証不能なACが存在する |
| R-2 | スコープ内/外が明記されているか | スコープ境界が不明瞭で実装範囲が確定できない |
| R-3 | NFR が漏れていないか（エラーハンドリング、セキュリティ、性能、環境分離） | エラーハンドリングのNFRが完全に欠落 |
| R-4 | 用語が定義されているか | 同一概念に複数の名称が混在 |
| R-5 | 機密情報の分類があるか（該当する場合） | 機密データを扱うのに分類・方針がない |

### Design

| # | チェック項目 | 致命的になる条件 |
|---|-------------|----------------|
| D-1 | 全ACをカバーする設計か（要件トレーサビリティ） | ACに対応するコンポーネント/処理が存在しない |
| D-2 | コンポーネントの入出力が型付きで定義されているか | 主要コンポーネントの入出力が未定義 |
| D-3 | 既存コードの再利用分析があるか | — (改善推奨止まり) |
| D-4 | アーキテクチャ図/シーケンス図があるか | — (改善推奨止まり) |
| D-5 | エラーシナリオが網羅されているか | 正常系しか記述がなく異常系が完全に欠落 |
| D-6 | 環境分離（dev/prod）が明示されているか | 環境差異があるのに未記述 |
| D-7 | パフォーマンス目標にコンポーネント別の内訳があるか | — (改善推奨止まり) |

### Test Design

| # | チェック項目 | 致命的になる条件 |
|---|-------------|----------------|
| T-1 | **全ACがカバレッジマトリクスに存在するか** | ACの漏れがある（最重要） |
| T-2 | テストレベル選択に根拠があるか | 根拠なくE2Eに偏重/Unit偏重 |
| T-3 | テストしない判断に理由があるか | 理由なしの空欄がある |
| T-4 | テストID体系が一貫しているか | — (軽微止まり) |

### Tasks

| # | チェック項目 | 致命的になる条件 |
|---|-------------|----------------|
| K-1 | テスト設計から逆算された順序か | テストより先に実装が来ている（TDD違反） |
| K-2 | 各タスクに Requirements トレースがあるか | トレースがないタスクが複数存在 |
| K-3 | 依存関係が DAG を形成しているか（循環なし） | 循環依存がある |
| K-4 | 手動タスクが明示的に分離されているか | 自動化タスクと手動タスクが混在し区別不能 |
| K-5 | Prompt フィールドが具体的か（Role / Task / Restrictions / Success） | Prompt が空またはワンライナー |

## フェーズ横断チェック（トレーサビリティ）

全フェーズが揃っている場合に実行する。一部フェーズが未作成の場合は、存在するフェーズ間のみチェックする。

| # | チェック項目 | 検出方法 |
|---|-------------|---------|
| X-1 | **孤立した要件** | requirements.md の FR/AC が design.md にマッピングされていない |
| X-2 | **テスト漏れ** | AC に対応するテストケースが test-design.md にない |
| X-3 | **追跡不能な設計** | design.md のコンポーネントに FR アンカーがない |
| X-4 | **タスク漏れ** | test-design.md のテストシナリオに対応するタスクが tasks.md にない |
| X-5 | **用語不一致** | フェーズ間で同じものを違う名前で呼んでいる |

**全てのトレーサビリティ漏れは「致命的」に分類する。**

## レビューレポートの出力形式

**原則:**
- AC追跡表は **問題のある行だけ** 出す。全行 OK のフェーズはカウントのみ
- トレーサビリティ指摘とフェーズ別指摘を **1本のリストに統合**（同じ問題の二重記載を防ぐ）
- テーブルではなく **見出し + 箇条書き** で出す（横幅問題の回避）

**出力ルール:**
- 指摘と指摘の間に `---` を入れて視覚的に分離する
- `---` の下は1行空ける
- 各指摘ブロック内は「対象 → 指摘内容 → 改善案」の3行構成。全て同じインデント

```markdown
# Spec Review: {spec-name}

致命的: X件 / 改善推奨: X件 / 軽微: X件
対象: Requirements, Design, Test Design, Tasks

（2〜3文。良い点と最大のリスクを端的に）
```

## 委譲モード（外部起動）

他スキルのサブエージェントとして起動される場合の入力仕様。
このモードでは Step 1（対象 spec の特定）をスキップし、呼び出し元が提供する入力に従う。

### 入力

| 項目 | 必須 | 説明 |
|------|------|------|
| spec_files | 必須 | spec ファイルの内容（requirements.md, design.md, test-design.md, tasks.md）。呼び出し元が取得方法を決定する |
| spec_name | 必須 | レポートの見出しに使う spec 名 |
| diff | 任意 | PR の diff テキスト。提供された場合、delta annotation を行う |

### 手順

1. spec_files で提供されたファイルを対象とする（Step 1 をスキップ）
2. Step 2〜5 を通常通り実行する
3. diff が提供されている場合、各 finding に delta ラベルをつける（判定基準は後述）
4. 通常のレビューレポート形式で出力する（delta ラベル付き）

### delta ラベリング判定基準

delta ラベルは finding の「原因となる要素」を diff が作成・変更したかどうかで判定する。

**フェーズ別 finding（単一ファイル内の指摘）:**

diff が指摘対象の FR/AC/コンポーネント/テストケースを含む → **[PR導入]**
diff が指摘対象の要素を含まない → **[既存]**

例:
- diff が FR-3 AC1 を変更 → R-1「AC1 が WHEN/THEN 形式でない」→ **[PR導入]**
- diff が FR-3 AC1 を変更 → R-1「FR-7 AC2 が WHEN/THEN 形式でない」→ **[既存]**

**トレーサビリティ finding（フェーズ横断の指摘）:**

トレーサビリティの指摘は「要素X → 要素Yの対応がない」という形を取る。
diff が要素X（トレース元）を作成・変更した → **[PR導入]**
diff が要素Xを作成・変更していない → **[既存]**

例:
- diff が FR-3 を変更 → X-1「FR-3 が design にマッピングされていない」→ **[PR導入]**
- diff が design に新コンポーネント追加 → X-4「コンポーネントに対応するタスクがない」→ **[PR導入]**
- diff が tasks.md のみ変更 → X-2「FR-5 のテスト漏れ」→ **[既存]**

**判定に迷ったら:** 「この PR がなければ、この finding は存在しなかったか？」— Yes なら [PR導入]、No なら [既存]。

### 出力

通常のレビューレポート形式。diff が提供された場合は delta ラベル付き。

## spec-writer との関係

| | spec-writer | spec-reviewer |
|---|---|---|
| 役割 | 書く | 読む |
| 対象 | 自分の spec | 他人の spec |
| 品質基準 | 執筆中のセルフチェック | 第三者レビュー |
| MCP | spec-workflow MCP を使用 | 使用しない（読み取りのみ） |
| 出力 | spec ファイル群 | レビューレポート |

品質基準の重複は意図的。spec-writer は書きながらのチェックリスト、spec-reviewer は完成物に対する検査。

## 自己評価（応答の最後に必ず実行）

レビューレポートを書き終えたら、送信前に以下を検証する。問題があればレポート自体を修正してから送信する。自己評価の結果はレポート末尾に簡潔に提示する。

### チェック1: 重要度の妥当性
- 致命的に分類した指摘は本当に「実装フェーズで手戻りが確実に発生する」レベルか
- 改善推奨を致命的に格上げしていないか。逆に致命的を改善推奨に格下げしていないか
- トレーサビリティ漏れを全て致命的に分類しているか（フェーズ横断チェックのルール）

### チェック2: トレーサビリティの網羅性
- requirements.md の全 FR/AC を走査したか。途中で打ち切っていないか
- 「追跡OK」と判定した FR について、本当に Design → Test → Task の全段を確認したか

### チェック3: 指摘の具体性
- 各指摘に「対象 → 指摘内容 → 改善案」の3要素が揃っているか
- 「〜が不足」「〜が曖昧」で止まっている指摘がないか。具体的に何が足りないか書いているか

### チェック4: 事実ベース
- spec ファイルに書かれていない内容を根拠にした指摘がないか
- 推測が混じっている場合「（要検証）」がついているか

### 提示ルール

- **問題を検出した項目**: 修正内容を明記する
- **問題なしの項目**: 一行で済ますか省略
- **分量は3〜5行**を目安。レビューレポートの末尾に独立セクションとして置く
