# Session Closer

セッション終了時に呼び出し、会話全体を分析して6カテゴリに抽出。Obsidian にフルアーカイブを保存し、次セッション開始用プロンプトを CLI に出力する。

## ワークフロー

```
session-closer 起動（wrap-up 経由 or 単独呼び出し）
  ├─ 1. 現在日時を取得
  ├─ 2. 会話コンテキスト全体を分析 → 6カテゴリ抽出
  ├─ 3. Obsidian にログ保存（6カテゴリ全量）
  ├─ 4. 次セッション用プロンプト生成（カテゴリから選択的抽出）
  └─ 5. 引き継ぎプロンプトを CLI に出力
```

対話なし。AskUserQuestion は使わない。呼ばれたら一発で完了する。

## Step 1: 日時取得

```bash
date "+%Y-%m-%d %H:%M:%S"
```

保存パス用: `YYYY/MM/DD/HHMMSS`

## Step 2: 6カテゴリ抽出

会話コンテキスト全体を読み、以下の6カテゴリに分類・抽出する。

### 2-1. コンテキスト

```markdown
- **プロジェクト**: {セッション主題を短いフレーズで}
- **対象**: {リポジトリ、ファイル、サービス名等}
- **開始状態**: {セッション開始時点の状況}
```

抽出元: セッション冒頭のユーザー発話 + 最初のツール操作対象

### 2-2. 成果・到達点

```markdown
- {意味のある作業単位での成果1}
- {成果2}
- **現在の状態**: {到達点の精密な記述}
```

抽出ヒューリスティック:
- **wrap-up の棚卸し結果が会話内に存在する場合、承認済みの Done をそのまま使う。session-closer は改変しない**
- wrap-up が未使用の場合、以下から自力抽出する:
  - ツール呼び出し履歴（Write, Edit, Bash 等の実行結果）
  - スキル使用結果
  - 「できた」「完了」「OK」系の発話周辺
  - ツール実行の羅列ではなく、意味のある作業単位でまとめる

### 2-3. 進行中の壁打ち

partner スキル（development-partner / design-partner / idea-breaker）がセッション中に使用された場合のみ抽出。使用されていなければ `（該当なし）` と明記。

複数の partner が使用された場合は、**最後にアクティブだった partner** の状態を記録する。フォワーディング履歴で遷移の経緯を補完する。

```markdown
- **使用スキル**: {development-partner / design-partner / idea-breaker}
- **フェーズ**: {下記フェーズ表から選択}
- **論点**: {何を決めようとしていたか / 何を構造化していたか}
- **選択肢**:
  - {選択肢1} ← 深掘り中 / 却下: {理由} / 採用: {理由}
  - {選択肢2}
- **フォワーディング履歴**: {スキル間の遷移。なければ省略}
```

フェーズ表:

| スキル | フェーズの選択肢 |
|---|---|
| development-partner | 問題構造化中 / 切り分け中 / フレーミング完了・フォワーディング待ち |
| design-partner | Discovery中 / 選択肢比較中 / 詳細設計中 / フォワーディング待ち |
| idea-breaker | 発散中 / 種の選別中 |

抽出ヒューリスティック:
- Skill ツールの呼び出し履歴（どの partner が起動されたか）
- AskUserQuestion の選択肢提示（比較表・選択肢の構造）
- 「案A」「案B」系の議論フロー
- フォワーディング提案の発話（「design-partner に渡す」等）
- フェーズ判定: 最後の partner 応答の内容から判断

該当なしなら `（該当なし）` と明記。

### 2-4. 意思決定

判断ごとにブロックで抽出。結論だけでなく思考の連鎖（判断チェーン）を記録する。

```markdown
#### D{N}: {論点}

- **問題認識**: {何を解こうとしていたか}
- **選択肢と評価**:
  - {選択肢A}: {評価} → **採用**: {理由}
  - {選択肢B}: {評価} → 却下: {理由}
- **決定**: {最終選択の要約}
- **影響**: {次セッション・実装への影響。セッション内で完結した場合は省略}
- **チェーン充足度**: 完全 / 部分的（欠落: {要素名}） / 抽出不可
```

チェーン充足度の判定基準:
- **完全**: 5要素（問題認識・選択肢列挙・評価・選択・却下理由）すべてが会話中に明示的な発話として存在
- **部分的**: 一部の要素が会話に存在しない。欠落要素を明記
- **抽出不可**: 結論のみ、または意思決定自体が会話から読み取れない。決定自体は記録する

抽出ヒューリスティック:
- AskUserQuestion の使用箇所（選択肢提示 → 回答の対）
- 「AにするかBにするか」系の議論 → 決定の流れ
- design-partner / development-partner の壁打ち中の判断ポイント
- 「それで」「そっちで」等の短い応答 → 直前の選択肢提示と対にして抽出
- 1ブロック = 1判断

該当なしなら `（該当なし）` と明記。

### 2-5. 未解決・Next Actions

**最重要カテゴリ。** 次セッションのアクションに直結する。

```markdown
1. **[高]** {未解決事項}: {コンテキスト・背景} → {issue URL}
2. **[中]** {未解決事項}: {コンテキスト・背景}
3. **[低]** {未解決事項}: {コンテキスト・背景}
```

優先度の判定基準:
- **[高]**: ブロッカー。これを解決しないと次に進めない
- **[中]**: 重要だが回避策がある、または次のスコープで対応予定
- **[低]**: nice-to-have。忘れないように記録

抽出ヒューリスティック:
- **wrap-up の棚卸し結果が会話内に存在する場合、承認済みの Follow-up をそのまま使う。session-closer は改変しない**
- **wrap-up が `gh issue create` を実行している場合、その出力から issue URL を取得し各項目に付記する**
- wrap-up が未使用の場合、以下から自力抽出する:
  - 明示的な TODO、「次は〜」「あとは〜」「残りは〜」系の発話
  - 着手したが完了しなかった作業
  - エラー未解決、ブロッカー
  - 議論したが結論が出なかった論点
  - TaskList に残っている pending/in_progress タスク

該当なしなら `（該当なし）` と明記。

### 2-6. 知見・地雷

3つのタグで分類:

```markdown
- **[知見]** {学んだこと}: {詳細}
- **[地雷]** {やってはいけないこと / ハマりポイント}: {理由・回避策}
- **[パターン]** {再現可能なプロセス}: {どこで使えるか}
```

抽出ヒューリスティック:
- **[知見]**: エラー解決で判明した事実、ライブラリの挙動、API仕様の発見、「わかった」「なるほど」系の発話周辺
- **[地雷]**: 方針転換（「やっぱり〜」「やめて〜」）、試して捨てたアプローチ、エラーの根本原因
- **[パターン]**: 複数回繰り返された手順、スキル間の連携パターン、「これ汎用化できそう」系の発話

該当なしなら `（該当なし）` と明記。

## Step 3: Obsidian 保存

```
ToolSearch: select:mcp__obsidian__write_note
```

### frontmatter

```yaml
---
date: "YYYY-MM-DD"
time: "HH:MM:SS"
project: "{コンテキストから推定したプロジェクト名}"
tags:
  - session-log
  - "{内容に応じたタグ1}"
  - "{内容に応じたタグ2}"
has_decisions: true/false
has_unresolved: true/false
has_issues: true/false
has_insights: true/false
has_active_partner: true/false
---
```

- `tags`: `session-log` は固定（後方互換）。他は内容に応じて 1〜3 個追加
- `has_*` フラグ: 該当カテゴリに実質的な内容がある場合のみ `true`。`（該当なし）` なら `false`
- `has_issues`: 未解決・Next Actions に issue URL が1件以上含まれる場合のみ `true`
- `has_active_partner`: partner スキルが使用中の場合のみ `true`

### 本文テンプレート

```markdown
## コンテキスト
- **プロジェクト**: {プロジェクト}
- **対象**: {リポジトリ、ファイル、サービス名}
- **開始状態**: {開始状態}

## 成果・到達点
- {成果1}
- {成果2}
- **現在の状態**: {到達点}

## 進行中の壁打ち
- **使用スキル**: {スキル名}
- **フェーズ**: {フェーズ}
- **論点**: {論点}
- **選択肢**:
  - {選択肢1}
  - {選択肢2}
- **フォワーディング履歴**: {履歴}

## 意思決定

### D1: {論点}
- **問題認識**: {問題認識}
- **選択肢と評価**:
  - {選択肢A}: {評価} → **採用**: {理由}
  - {選択肢B}: {評価} → 却下: {理由}
- **決定**: {決定}

1. **[高]** {未解決事項}: {コンテキスト} → {issue URL}

```

- パス: `session-logs/YYYY/MM/DD/HHMMSS.md`
- モード: 新規作成
- 未解決・Next Actions に issue URL がある場合（wrap-up が `gh issue create` を実行済み）、各項目に付記する

## Step 4: 次セッション用プロンプト生成

6カテゴリから選択的に抽出し、次セッション開始用のプロンプトを組み立てる。

### 抽出ルール

| カテゴリ | プロンプトへの抽出 |
|---|---|
| コンテキスト | 全量 |
| 成果・到達点 | 全量（「現在の状態」を重視） |
| 進行中の壁打ち | `has_active_partner=true` の場合のみ全量。`（該当なし）` ならセクションごと省略 |
| 意思決定 | `影響` フィールドに実質的内容があるもののみ（影響が省略されたブロックは除外） |
| 未解決・Next Actions | 全量（issue URL があれば含める） |
| 知見・地雷 | `[地雷]` タグのみ |

### プロンプトテンプレート

```markdown
前回のセッション（YYYY-MM-DD HH:MM:SS）の続き。

## コンテキスト
- **プロジェクト**: {プロジェクト}
- **対象**: {リポジトリ、ファイル、サービス名}
- **開始状態**: {開始状態}

## 成果・到達点
- {成果1}
- {成果2}
- **現在の状態**: {到達点}

## 進行中の壁打ち
- **使用スキル**: {スキル名}
- **フェーズ**: {フェーズ}
- **論点**: {論点}
- **選択肢**:
  - {選択肢1}
  - {選択肢2}
- **フォワーディング履歴**: {履歴}

## 意思決定

### D1: {論点}
- **問題認識**: {問題認識}
- **選択肢と評価**:
  - {選択肢A}: {評価} → **採用**: {理由}
  - {選択肢B}: {評価} → 却下: {理由}
- **決定**: {決定}
- **影響**: {影響}
- **チェーン充足度**: {充足度}

## 未解決・Next Actions
1. **[高]** {未解決事項}: {コンテキスト} → {issue URL}
2. **[中]** {未解決事項}: {コンテキスト}

## 知見・地雷
- **[地雷]** {地雷}: {理由・回避策}
```

### プロンプト出力ルール

- 該当なしのセクションは**セクションごと省略**。プロンプトは短いほど良い
- 「未解決・Next Actions」が空の場合のみ、省略せず `（なし — 前回セッションで完結）` と明記
- 各セクション最大5項目に絞る

## Step 5: CLI 出力

Step 4 で生成したプロンプトを CLI に出力する。

```
---
Session Log saved to Obsidian: session-logs/YYYY/MM/DD/HHMMSS.md
---

以下を次のセッションの最初のメッセージとして貼り付けてください:

---

{プロンプト本文}

---
```

## やってはいけないこと

- **対話しない**。AskUserQuestion を使わない。呼ばれたら黙って記録・出力して終わる
- **評価・採点しない**。「良いセッションだった」等の価値判断を入れない
- **Obsidian 側のセクションを省略しない**。該当なしでも `（該当なし）` と明記
- **プロンプトに不要な情報を入れない**。`[知見]` `[パターン]` はアーカイブ専用。プロンプトには `[地雷]` のみ
- **会話の要約を書かない**。カテゴリ別の構造化された抽出を行う
- **ユーザーの発話を改変しない**。意思決定の根拠等はユーザーの言葉を尊重する
- **フラグを推測で true にしない**。実質的な内容がなければ `false`
- **partner 未使用時に「進行中の壁打ち」を推測で埋めない**。Skill ツールの呼び出し履歴に partner スキルがなければ `（該当なし）`
- **優先度を根拠なく付けない**。未解決事項の優先度はセッション中の発話・文脈から判断する
- **wrap-up の承認済み棚卸しを改変しない**。wrap-up が会話内で棚卸し・承認を行っている場合、成果・到達点と未解決・Next Actions はその結果をそのまま使う

## 自己評価（応答の最後に必ず実行）

Obsidian 保存・CLI 出力の前に以下を検証する。問題があれば出力内容を修正してから出力する。自己評価の結果は Obsidian ログには含めない（セッション記録の関心事ではない）。

### チェック1: has_* フラグの整合性

- 各 `has_*` フラグが、対応するカテゴリの実質的内容の有無と一致しているか？
- `（該当なし）` なのに `true` になっていないか？ 内容があるのに `false` になっていないか？
- `has_issues` が `true` なのに issue URL が記録されていないか？

### チェック2: 推測混入チェック

- 抽出した内容は、すべて会話中の発話・ツール実行結果に根拠があるか？
- 会話に存在しない情報を補完していないか？
- 特に意思決定のチェーン充足度: 会話に存在しない要素を「完全」に含めていないか？

### チェック3: カテゴリ間の重複・漏れ

- 同じ情報が複数カテゴリに重複して記載されていないか？（特に「進行中の壁打ち」の選択肢と「意思決定」の判断チェーンの重複）
- 会話中の重要なトピックが、どのカテゴリにも抽出されていない漏れはないか？

### チェック4: 出力間の整合性

- Obsidian ログと引き継ぎプロンプトの2出力間で、同じ判断に対して矛盾する記述がないか？
- プロンプトの抽出ルール（影響フィールドの有無、地雷タグのみ等）が正しく適用されているか？

### チェック5: 次セッション用プロンプトの実用性

- プロンプトだけで次セッションを開始できる情報量があるか？
- 不要なセクションが省略されているか？（該当なしセクションの混入）
- 各セクション最大5項目の制限を超えていないか？

### チェック6: wrap-up 棚卸しとの整合性

- wrap-up が会話内で棚卸し・承認を行っている場合、成果・到達点と未解決・Next Actions がその承認済み結果と一致しているか？
- wrap-up が `gh issue create` を実行している場合、issue URL が未解決・Next Actions に正しく紐づいているか？

### 提示ルール

- CLI 出力ブロックの**直前**に独立セクションとして提示する
- **問題を検出した項目**: 修正内容を明記する
- **問題なしの項目**: 省略
- **全項目問題なし**: 「自己評価: 全チェック通過」の1行のみ
- **分量は1〜3行**を目安

## Tools Required

- `Bash`: 日時取得（`date` コマンド）
- `mcp__obsidian__write_note`: Obsidian 保存
