## Linux Kernel

### 核心的設計判断
- **モノリシックカーネル** を選択（Minix の Tanenbaum とのマイクロカーネル論争）
  - 理由: パフォーマンス。コンテキストスイッチのコストを避ける
  - ただしモジュール機構（Loadable Kernel Module）で柔軟性を確保
- **Everything is a File** — デバイス、プロセス情報（/proc）、ネットワークソケットまでファイルとして抽象化
  - 統一インターフェースの力。全てが read/write/open/close で操作できる

### 引用すべき設計教訓
- 「早すぎる最適化は悪だが、アーキテクチャレベルの性能設計は最初にやるべき」
- 統一的な抽象化（File Descriptor）が複雑なシステムを扱いやすくする
- モノリス vs マイクロの議論は「理論的な優雅さ」より「実際のパフォーマンス特性」で決める

## Git

### 核心的設計判断
- **Content-addressable storage** — SHA-1 ハッシュでオブジェクトを管理
  - blob, tree, commit, tag の4種のオブジェクト
  - 同じ内容は同じハッシュ → 自然な重複排除
- **分散型** — 全クローンが完全な履歴を持つ。中央サーバーは不要
- **スナップショット** モデル（差分ではなく、各コミットがツリー全体を指す）
  - ブランチは単なるポインタ（40バイトのファイル）。ブランチ作成が O(1)
- **Merkle Tree** — ツリーのハッシュが子のハッシュを含む。改ざん検知が自動

### 引用すべき設計教訓
- データモデルが正しければ、その上の操作は自然に正しくなる
- 「ブランチが安い」はデータ構造の設計から来ている（ポインタ1つ）
- Immutable なデータ構造は並行処理と整合性検証を劇的に簡素化する

## PostgreSQL

### 核心的設計判断
- **MVCC (Multi-Version Concurrency Control)** — 読み取りがロックを取らない
  - 各行に xmin/xmax（トランザクションID）を持ち、可視性を判定
  - VACUUM が古いバージョンを回収（PostgreSQL 固有の設計トレードオフ）
- **拡張性アーキテクチャ** — カスタム型、関数、演算子、インデックス手法を追加可能
  - PostGIS（地理情報）、pgvector（ベクトル検索）が良い例
- **WAL (Write-Ahead Logging)** — 変更をまずログに書き、後でデータファイルに反映
  - クラッシュリカバリ、レプリケーションの基盤

### 引用すべき設計教訓
- 拡張性を設計に組み込むと、予想外のユースケースに対応できる
- MVCC のように「読み書き分離」を低レイヤーで実現すると、上位の設計が簡素化される
- トレードオフは消えない。MVCC の代償が VACUUM 問題

## Apache Kafka

### 核心的設計判断
- **Append-only Log** — メッセージは追記のみ。削除しない（retention で制御）
  - シーケンシャル I/O で圧倒的スループット
- **Consumer Group** — パーティション単位でコンシューマに割り当て
  - スケーラビリティと順序保証のバランス（パーティション内は順序保証）
- **Zero-copy** — カーネルの sendfile でデータを直接 NIC に送る。CPU コピーを回避
- **Pull ベース** — Consumer が自分のペースで取得。Push だとコンシューマの処理速度に合わせる必要がある

### 引用すべき設計教訓
- シンプルなデータ構造（ログ）の上に複雑な機能を構築する
- 「ログ」はただのデータ構造ではなく、分散システムの根幹概念（Jay Kreps "The Log"）
- パフォーマンスは OS の仕組み（ページキャッシュ、sendfile）を活かす設計から生まれる

## Redis

### 核心的設計判断
- **シングルスレッド + イベントループ** — ロック不要。データ構造操作がアトミック
  - 6.0 以降 I/O はマルチスレッド化（パース・レスポンス書き出し）だがコマンド実行はシングル
- **豊富なデータ構造** — String, Hash, List, Set, Sorted Set, Stream, HyperLogLog
  - 「データ構造サーバー」。キャッシュ以外の用途（キュー、ランキング、セッション等）を自然に実現
- **RDB + AOF** の2つの永続化戦略。ユースケースで選択・併用可能

### 引用すべき設計教訓
- シングルスレッドは制約ではなく設計判断。複雑さを排除してパフォーマンスを得る
- 適切なデータ構造の選択がアルゴリズムより重要なことがある
- 「一つのことをうまくやる」（Sorted Set がランキング、Rate Limiting、優先度キューを全て解決）

## Nginx

### 核心的設計判断
- **イベント駆動 + 非同期 I/O** — Apache の「リクエストごとにプロセス/スレッド」モデルとの対比
  - C10K問題の解決。少ないリソースで大量の同時接続を処理
- **Master-Worker アーキテクチャ** — Master がシグナルを受け、Worker が実際の処理
  - Graceful reload: 新 Worker を起動し、旧 Worker は処理中のリクエストを完了してから終了

### 引用すべき設計教訓
- リソースモデルの選択（スレッド per リクエスト vs イベントループ）がスケーラビリティを決定する
- C10K 問題: 「スレッドモデルはスケールしない」を証明し、イベント駆動パラダイムを推進した

## SQLite

### 核心的設計判断
- **サーバーレス** — ライブラリとしてアプリに組み込む。プロセス間通信ゼロ
- **単一ファイル** — データベース全体が1ファイル。コピー = バックアップ
- **テスト密度** — 100% ブランチカバレッジ。テストコードが本体の661倍（公式発表）

### 引用すべき設計教訓
- 「サーバーが本当に必要か？」を問う設計判断
- シンプルさは機能ではなく前提条件。SQLite が最も使われている DB である理由
- テストの徹底が「組み込みで使われる信頼性」を生んでいる

## Unix Philosophy

### 核心原則
1. **一つのことをうまくやるプログラムを書け**
2. **協調して動くプログラムを書け**（パイプ、テキストストリーム）
3. **テキストストリームを普遍的インターフェースとして使え**

### 設計への示唆
- 小さな道具の組み合わせ → マイクロサービスの思想的源流
- パイプ = メッセージパッシング。stdin/stdout = 標準化されたインターフェース
- 「テキスト」という最もシンプルなフォーマットが最も柔軟

### 現代への適用
- CLI ツール設計: 入力は stdin、出力は stdout、設定は環境変数
- API 設計: 小さなエンドポイント、組み合わせ可能、テキスト（JSON）ベース
- マイクロサービス: 「一つのことをうまくやる」サービス群を連携させる

## AWS の設計原則

### Well-Architected Framework の柱
1. **運用上の優秀性**: 自動化、継続的改善
2. **セキュリティ**: 最小権限、多層防御
3. **信頼性**: 障害からの自動復旧、水平スケール
4. **パフォーマンス効率**: 適切な技術選択、実験の容易さ
5. **コスト最適化**: 消費モデル、効率測定
6. **サステナビリティ**: リソース効率の最大化

### Amazon のアーキテクチャ原則（Werner Vogels）
- **Everything fails all the time** — 障害を前提に設計
- **Design for failure** — Circuit Breaker, Retry, Graceful Degradation
- **API-first** — 全てのサービスはAPIで公開（Bezos Mandate, 2002年）
- **Two-Pizza Teams** — チームサイズがアーキテクチャを規定する（Conway's Law の実践）
