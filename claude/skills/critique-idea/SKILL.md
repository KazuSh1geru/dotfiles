# Critique Idea

入力されたアイデアに対して、批判的な視点から分析を行い、盲点・未検証の前提・弱点を明らかにする。

## 入力

- `$ARGUMENTS`: 分析対象のファイルパス、またはアイデアの概要テキスト
- ファイルパスの場合は内容を読み取る

## 出力先

- 単独実行: 入力ファイルと同じディレクトリに `[アイデア名]-critique.md`
- `/idea-feedback` から呼び出し: 親スキルが指定するディレクトリに出力
  - ソリューション仮説: `ideas/solutions/[アイデア名]/[アイデア名]-critique.md`

## Workflow

Multi-Agent Critical Thinking Framework（EDU-Prompting）を参考に、4段階で分析を行う：

### Phase 0: ユーザーの声収集（エビデンス準備）
3つのデータソースから声を収集:

- `/query-user-voice` - 従業員サーベイから関連する声を検索
- `/query-hr-voice` - 人事・経営ヒアリングから関連するニーズを検索（**個社情報は出力しない**）
- `/query-talks` - 過去の会議録から関連する議論・発言を検索

収集した声は以下のフェーズでエビデンスとして活用する。

### Phase 1: Context Gathering（文脈の整理）
- アイデアの前提を列挙
- ターゲット・スコープを明確化
- 暗黙の仮定を顕在化
- **ユーザーの声から見える実態との照合**

### Phase 2: Validity Check（妥当性検証）
- 各前提は検証可能か？
- 検証済みの前提と未検証の前提を分離
- 致命的な前提の特定
- **ユーザーの声で検証できる前提を特定し、引用で裏付け**

### Phase 3: Critique（批評）
- 各観点からの批判
- 反論の構築
- 弱点の深掘り

### Phase 4: Meta-Review（統合）
- 批判の優先度付け
- 対処可能性の評価
- 次のアクションの提案

## 批評観点チェックリスト

### 課題定義について
- [ ] 本当にその課題は存在するか？（エビデンス）
- [ ] 課題の規模は十分か？（市場性）
- [ ] 課題を抱えているのは本当にそのセグメントか？
- [ ] より本質的な課題が隠れていないか？
- [ ] 課題は解決されるべきか、それとも受容されるべきか？

### ソリューションについて
- [ ] このアプローチでなければならない理由は？
- [ ] よりシンプルな解決策はないか？
- [ ] スケールするか？
- [ ] 持続可能か？
- [ ] 誰かが既に試して失敗していないか？

### ターゲットについて
- [ ] ターゲットは明確に定義されているか？
- [ ] ターゲットは実際に到達可能か？
- [ ] ターゲットは支払い意思があるか？
- [ ] より適切なターゲットはいないか？

### 実現性について
- [ ] 技術的に実現可能か？
- [ ] リソース（時間・人・金）は足りるか？
- [ ] 規制・法的な障壁はないか？
- [ ] 依存する外部要因は何か？

### 差別化について
- [ ] 既存ソリューションとの違いは明確か？
- [ ] その違いはユーザーにとって価値があるか？
- [ ] 競合に真似されたらどうなるか？
- [ ] 参入障壁は構築できるか？

## 出力フォーマット

```markdown

> — メンバー, 勤続X年, XX歳

**人事・経営ヒアリングから**:

**過去の会議から**:
> 「[関連する発言の引用]」

2. [前提2]

#### 暗黙の前提（顕在化）
1. [暗黙の前提1]
2. [暗黙の前提2]

### Phase 2: 妥当性検証

| 前提 | 検証状況 | リスク |
|-----|---------|-------|
| [前提1] | 検証済み/未検証/検証不能 | 高/中/低 |

#### 致命的な未検証前提
- [最も危険な前提とその理由]

### Phase 3: 批評

#### 最大の弱点
[アイデアの最も大きな弱点]

#### 見落としている観点
1. [観点1]
2. [観点2]

#### Devil's Advocate（悪魔の代弁者）
[このアイデアが失敗するシナリオ]

### Phase 4: 統合

#### 批判の優先度
1. **致命的**: [すぐに対処が必要]
2. **重要**: [対処すべき]
3. **注意**: [認識しておくべき]

#### 対処可能性
| 批判点 | 対処可能性 | 対処案 |
|-------|-----------|-------|
| [批判1] | 可能/困難/不可能 | [案] |

#### 推奨アクション
1. [最優先で検証すべきこと]
2. [次に取り組むべきこと]
```

## 注意事項

- 批判は建設的に。破壊ではなく改善が目的
- 「それでもこのアイデアを進めるなら」という視点を忘れない
- 全否定ではなく、条件付きで成立する場合も検討
- 批判者の役割に徹する（自分の好みを入れない）
