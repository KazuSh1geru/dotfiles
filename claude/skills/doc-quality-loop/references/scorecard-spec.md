# スコアカード v1（spec）

仕様書・設計書の品質を判定するための評価基準。
Generator が出力したドラフトに対し、Reviewer（spec-reviewer・批判的読者）がこのスコアカードで判定する。

## MUST（全通過必須）

すべて PASS でなければ収束しない。1つでも FAIL なら改善ループを継続する。

| ID | 項目 | 判定基準 | FAIL の典型例 |
|---|---|---|---|
| M1 | 要件とタスクのトレーサビリティ | Requirements → Tasks の追跡が可能。各タスクがどの要件に対応するか明示されている | タスクが列挙されているが、どの要件から導出されたか不明 |
| M2 | 完了条件が検証可能 | 各タスク・要件の完了条件が具体的で検証可能。「適切に」「十分に」「概ね」が使われていない | 「適切にテストする」「十分な品質を確保する」で終わる |
| M3 | スコープが明確 | やること（In Scope）とやらないこと（Out of Scope）が定義されている | スコープの記述がない。または「やること」だけで「やらないこと」が未定義 |
| M4 | 前提条件・制約が明示 | 技術的前提、ビジネス制約、依存関係が明記されている | 前提条件のセクションがない。暗黙の前提に依存している |
| M5 | 各セクションが自己完結 | セクション内で必要な情報が完結している。前方参照（「後述する〜」）がない | 「詳細は後述」が繰り返される。セクション間の循環参照がある |

## WANT（改善推奨）

MUST を全通過した上で、品質向上のために判定する。

| ID | 項目 | 判定基準 | YES の例 |
|---|---|---|---|
| W1 | エッジケースが考慮されている | 正常系だけでなく異常系・境界値・例外ケースが記載されている | 「〇〇が空の場合」「タイムアウト時の挙動」等の記述がある |
| W2 | 代替案とトレードオフが記載 | 選択した設計の代替案と、なぜ現案を選んだかの理由が明記されている | 「案A vs 案B: 〇〇の理由で案Aを採用」の記述がある |
| W3 | 具体的な数値・期限がある | 性能要件、SLA、マイルストーン等に具体的な数値が含まれている | 「レスポンスタイム 200ms 以下」「3月末までに完了」 |
| W4 | 図・表で構造が視覚化 | アーキテクチャ図、シーケンス図、テーブル定義等が含まれている | Mermaid図、テーブル、フローチャートが含まれている |
| W5 | レビュアーが判断に必要な情報が過不足なく含まれている | 不足情報で判断が止まらず、かつ冗長な情報で読み手の負荷が上がっていない | 意思決定に必要な情報がすべて揃い、不要な詳細は付録に退避されている |

## 収束条件

```
IF MUST 全項目 PASS:
  IF ループ回数 >= 2 OR WANT の YES が 3/5 以上:
    → 完了。最終ドラフトを出力
  ELSE:
    → WANT 改善のためもう1ループ
ELSE:
  IF ループ回数 >= 5:
    → 打ち切り。残存課題レポートを人間に提示
  ELSE:
    → MUST FAIL 項目の改善指示を出して次ループ
```

## 判定出力フォーマット

Reviewer は以下の形式で判定を出力する:

```markdown
### MUST 判定

| ID | 判定 | 根拠 | 該当箇所 |
|---|---|---|---|
| M1 | PASS/FAIL | （具体的な根拠） | （ドラフト内の該当箇所を引用） |
| M2 | PASS/FAIL | （具体的な根拠） | （該当箇所） |
| M3 | PASS/FAIL | （具体的な根拠） | （該当箇所） |
| M4 | PASS/FAIL | （具体的な根拠） | （該当箇所） |
| M5 | PASS/FAIL | （具体的な根拠） | （該当箇所） |

### WANT 判定

| ID | 判定 | 根拠 | 該当箇所 |
|---|---|---|---|
| W1 | YES/NO | （具体的な根拠） | （該当箇所） |
| W2 | YES/NO | （具体的な根拠） | （該当箇所） |
| W3 | YES/NO | （具体的な根拠） | （該当箇所） |
| W4 | YES/NO | （具体的な根拠） | （該当箇所） |
| W5 | YES/NO | （具体的な根拠） | （該当箇所） |
```
