# スコアカード v1（general）

汎用ドキュメント（提案書・ノート・スライド・レポート等）の品質を判定するための評価基準。
Generator が出力したドラフトに対し、Reviewer（target-reader・批判的読者）がこのスコアカードで判定する。

## MUST（全通過必須）

すべて PASS でなければ収束しない。1つでも FAIL なら改善ループを継続する。

| ID | 項目 | 判定基準 | FAIL の典型例 |
|---|---|---|---|
| M1 | 目的と対象読者が冒頭で明確 | 誰に何を伝えるかが冒頭で明示されている。読者がこのドキュメントを読む理由がわかる | 目的の記述がない。「〜について」で始まり誰向けか不明 |
| M2 | 主張に根拠がある | 「〜と思う」「〜な気がする」で終わらず、データ・事例・論理による裏付けがある | 「おそらく〜だろう」「〜と思われる」で終わる主張が複数ある |
| M3 | 論理の流れが自然 | 前提→主張→根拠→結論の順で構成されている。論理の飛躍がない | 前提の説明なく結論が提示される。主張と根拠が対応していない |
| M4 | アクションまたは結論が明確 | 読了後に何が変わるか（次のアクション / 意思決定 / 理解）が明確 | 「〜についてまとめました」で終わり、So What が不在 |
| M5 | セクション間に論理の飛躍がない | 各セクションが論理的に接続されており、読者が迷子にならない | セクション間の繋がりが不明。突然別の話題が始まる |

## WANT（改善推奨）

MUST を全通過した上で、品質向上のために判定する。

| ID | 項目 | 判定基準 | YES の例 |
|---|---|---|---|
| W1 | 1文1メッセージ | 各記述が1文に1つのメッセージのみ含む。接続詞で複数要因を繋いでいない | 短く分割された箇条書き。「〜し、〜で、〜が」の三重接続がない |
| W2 | AI的定型表現の排除 | 「〜を推進する」「〜を強化する」「〜を図る」等の抽象動詞・定型文がない | 具体的な動詞（「書く」「実行する」「確認する」「変更する」）が使われている |
| W3 | トレードオフ・制約の提示 | メリットだけでなくデメリット・制約・リスクが言及されている | 「ただし、〇〇の制約がある」「〇〇のリスクを受け入れる前提で」 |
| W4 | 具体例・数値による裏付け | 抽象的な主張が具体例や数値で補強されている | 「例えば〇〇の場合」「過去3ヶ月で〇〇件」等の具体的記述 |
| W5 | 読者の前提知識が明示 | 読者が何を知っていれば読めるドキュメントか明示されている | 「この提案は〇〇プロジェクトの背景を知っている方を想定しています」 |

## 収束条件

```
IF MUST 全項目 PASS:
  IF ループ回数 >= 2 OR WANT の YES が 3/5 以上:
    → 完了。最終ドラフトを出力
  ELSE:
    → WANT 改善のためもう1ループ
ELSE:
  IF ループ回数 >= 5:
    → 打ち切り。残存課題レポートを人間に提示
  ELSE:
    → MUST FAIL 項目の改善指示を出して次ループ
```

## 判定出力フォーマット

Reviewer は以下の形式で判定を出力する:

```markdown
### MUST 判定

| ID | 判定 | 根拠 | 該当箇所 |
|---|---|---|---|
| M1 | PASS/FAIL | （具体的な根拠） | （ドラフト内の該当箇所を引用） |
| M2 | PASS/FAIL | （具体的な根拠） | （該当箇所） |
| M3 | PASS/FAIL | （具体的な根拠） | （該当箇所） |
| M4 | PASS/FAIL | （具体的な根拠） | （該当箇所） |
| M5 | PASS/FAIL | （具体的な根拠） | （該当箇所） |

### WANT 判定

| ID | 判定 | 根拠 | 該当箇所 |
|---|---|---|---|
| W1 | YES/NO | （具体的な根拠） | （該当箇所） |
| W2 | YES/NO | （具体的な根拠） | （該当箇所） |
| W3 | YES/NO | （具体的な根拠） | （該当箇所） |
| W4 | YES/NO | （具体的な根拠） | （該当箇所） |
| W5 | YES/NO | （具体的な根拠） | （該当箇所） |
```
